# LanceDB Self-Study Course

A 4–5 day, ~4 hours/day course to take you from "knows Rust basics" to "can read,
debug, and contribute to LanceDB's Rust core and its Python/TypeScript bindings".

It is designed to be read **in order**, alongside the actual source tree. Every
chapter ends with **what to read in the repo**, **exercises**, and a list of
**confirmation questions** you should be able to answer before moving on.

---

## What LanceDB actually is (one screen)

Read `README.md` and `AGENTS.md` at the repo root first. The 30-second summary:

* LanceDB is a **vector + full-text + SQL retrieval database**.
* It is an **embedded library**, not a server. You link `lancedb` as a Rust crate
  (or its Python / Node / Java bindings) and it manages files on disk or in
  object storage. Think "SQLite, but for vectors".
* It's a **wrapper around the [Lance](https://github.com/lance-format/lance)
  columnar format**. LanceDB does *catalog* (databases, tables, indexes, embedding
  pipelines, query/insert ergonomics, remote support). Lance does the *storage
  format* (manifests, fragments, encodings, vector index implementations).
* Two backends:
    * **Local** ("OSS"): a `Database` that lists Lance datasets on a filesystem
      or object store. In-process, like SQLite.
    * **Remote**: an HTTP client that talks to LanceDB Cloud. Same `Table` /
      `Connection` API, different transport.
* Public surface: `lancedb::connect(...)` → `Connection` → `create_table` /
  `open_table` → `Table` → `query()` / `vector_search()` / `add()` / `update()`
  / `create_index()`.

Repo layout (from `AGENTS.md`):

```
rust/lancedb        Rust core         <-- 90% of your reading time
python              PyO3 bindings + Python wrapper
nodejs              napi-rs bindings + TypeScript wrapper
java                JNI bindings (we will skim only)
docs                user-facing docs
```

## How to use this course

1. **Open the repo in your editor and the chapter side-by-side.** Every chapter
   tells you exactly which `.rs` files to open. Don't try to read them in
   isolation — the value is in jumping into source as you read.
2. **Compile early.** From the repo root:
   ```
   cargo check --quiet --features remote --tests --examples
   ```
   That gets cargo to download all dependencies and build the index your IDE
   uses. The first run is slow (~5 min); after that, "go to definition" works.
3. **Run the simple example on Day 1.** It exercises ~80% of the public API:
   ```
   cargo run --example simple
   ```
   File: `rust/lancedb/examples/simple.rs`.
4. **Use the AGENTS.md commands.** They are the canonical way to test/lint:
   ```
   cargo test  --quiet --features remote --tests
   cargo clippy --quiet --features remote --tests --examples
   cargo fmt --all
   ```
5. **Take notes.** Each chapter has a "questions to be able to answer" list;
   write one or two sentences for each. If you can't, re-read.

## How I (the assistant) can accelerate this beyond reading

Ask me to do any of these as you go — they're high-leverage and cheap:

1. **"Trace this call for me."** Pick a public method (e.g. `Table::add`,
   `Connection::create_table`, `Query::nearest_to`) and I'll walk you from the
   public API down through the trait, the local impl, the Lance call, and the
   remote impl, all in one message.
2. **"Summarize this file."** For any file > 500 lines (e.g. `table.rs` is 146k
   bytes), I can produce a "what each section does" outline so you don't get
   lost.
3. **"Diff: local vs remote."** Pick a method; I'll show the local
   (`NativeTable`) and remote (`RemoteTable`) implementations side-by-side and
   call out the differences.
4. **"Explain this Rust idiom."** When you hit `async_trait`, `Arc<dyn Trait>`,
   `pin-project`, `snafu`, builder pattern with `mut self`, GATs, `impl Trait
   in argument`, etc. — paste the snippet and ask, I'll explain in context.
5. **"Generate exercises at my level."** If a chapter feels too easy or too
   hard, ask me for harder/easier exercises that touch the same files.
6. **"Spot-check my mental model."** Tell me what you think a function does;
   I'll check it against the source and correct anything wrong.
7. **"Help me make a one-line change."** When you do Day 5, I can pair with you
   on a real PR: rename a method, add a metric, fix a typo in a doc comment.
   I will *not* push for you, but I will write tests and walk you through
   review-ready changes.
8. **"Index search."** Need to know "where in the codebase do we use moka?" or
   "what calls `BaseTable::create_plan`?" — ask. Faster than grep when you
   don't know the right keyword yet.

## Prerequisites checklist

Before Day 1, make sure:

* [ ] Rust toolchain installed (the repo pins a version in `rust-toolchain.toml`
      — `rustup` will pick it up automatically when you `cd` into the repo).
* [ ] `cargo check --quiet --features remote --tests --examples` succeeds at
      the repo root. (First run downloads ~1 GB of crates and builds them.)
* [ ] `cargo run --example simple` succeeds and prints a `RecordBatch` to your
      terminal.
* [ ] Your editor has rust-analyzer set up (VS Code: "rust-analyzer" extension;
      JetBrains: built-in). "Go to definition" must work — you will use it
      constantly.
* [ ] Optional but very useful: `cargo install cargo-expand` (for inspecting
      what `#[derive(...)]` and `#[async_trait]` macros expand to).

If any of these fail, fix them first; don't push through. The whole course
assumes you can navigate code with your editor.

## What you will be able to do at the end

* Read any file in `rust/lancedb/src/` without getting lost.
* Trace a single API call from `lancedb` (Rust), `python/python/lancedb`
  (Python), or `nodejs/lancedb` (TypeScript) all the way down to the Lance
  dataset (or the HTTP request, in the remote case).
* Understand what each major dependency (`arrow`, `datafusion`, `lance`,
  `tokio`, `object_store`, `snafu`, `pyo3`, `napi-rs`) does for LanceDB.
* Add a small new method on `Table` end-to-end across Rust core, Python, and
  TypeScript, with tests, following the recipe in `AGENTS.md`.
* Confidently file a "good first issue" PR.

Open `01_rust_essentials_for_lancedb.md` to begin.
