# Day 3 — lance-encoding + lance-file (~4 hours)

**Goal:** by the end of today you understand how an Arrow `RecordBatch` becomes bytes on disk and back. You can read the encoder/decoder traits, recognize logical vs physical encodings, and you can navigate a `.lance` file's footer.

This is the densest day. The encoding layer is where Lance differs most from Parquet — random-access friendliness, blob support, and the "structural encoding" model are all here.

## Block 1 — `lance-encoding`: the model (75 min)

### 1.1 The big picture (10 min)

Lance has had a few file format versions:

- **Legacy / 0.1** — early format, kept for read compatibility only.
- **2.0** — first version of the "structural encoding" model. Protos in [`protos/encodings_v2_0.proto`](../protos/encodings_v2_0.proto).
- **2.1** (current default) — refined structural encodings. Protos in [`protos/encodings_v2_1.proto`](../protos/encodings_v2_1.proto).
- **2.2** — incremental, mostly stable.
- **2.3** — unstable / experimental.

See [`rust/lance-encoding/src/version.rs`](../rust/lance-encoding/src/version.rs):

```rust
pub enum LanceFileVersion {
    Legacy, V2_0, #[default] V2_1, Stable, V2_2, Next, V2_3,
}
```

When you read or write, you pick a version. New datasets default to `V2_1` (because `Stable` resolves to `V2_1`). The legacy format gets its own readers in `lance-file/src/previous/`.

For the curriculum, treat **2.1 as "the format"** and only acknowledge legacy where needed.

### 1.2 The two phases: scheduling and decoding (15 min)

Open [`rust/lance-encoding/src/decoder.rs`](../rust/lance-encoding/src/decoder.rs) and read **the module-level doc comment** at the top (lines 1–80). This is one of the best pieces of documentation in the codebase. Internalize:

- **Reading is two phases: scheduling and decoding.**
  - *Scheduling* figures out which byte ranges to fetch and in what order. It's I/O-bound.
  - *Decoding* turns those raw bytes into Arrow arrays. It's CPU-bound.
- **There are two granularities of decoder:**
  - `LogicalPageDecoder` — decodes one or more columns of Lance data into an Arrow array. One per "Arrow field."
  - `PhysicalPageDecoder` — decodes a single column of bytes. Doesn't have to produce Arrow output (so it can stay in efficient native representations like bit-packed integers).
- **Encodings nest into trees.** A `List<UInt32>` has a logical "list" encoder which contains a logical "primitive" encoder for offsets and another for items, each of which contains physical encodings (basic, RLE, value, validity bitmap…) underneath.

The doc comment shows this with a worked example. Read it twice.

### 1.3 The `EncodingsIo` trait (5 min)

Re-open [`rust/lance-encoding/src/lib.rs`](../rust/lance-encoding/src/lib.rs) and re-read it. The `EncodingsIo` trait is the contract:

```rust
pub trait EncodingsIo: std::fmt::Debug + Send + Sync {
    fn submit_request(&self, ranges: Vec<Range<u64>>, priority: u64)
        -> BoxFuture<'static, Result<Vec<Bytes>>>;
}
```

Key points:

- **Priority matters.** Lower priority = "I want this sooner." This sounds backwards but maps to "row number" — earlier rows are higher priority because the user is consuming the stream in order.
- **Coalescing happens above this trait.** The implementor (e.g., `ScanScheduler`) groups nearby ranges into one HTTP request, but still returns one `Bytes` per requested range.
- **`with_bypass_backpressure`** is for indirect I/O — fetching items after decoding offsets, where blocking would deadlock.

`BufferScheduler` (also in this file) is a trivial in-memory implementation used in tests.

### 1.4 The encoder side — `encoder.rs` (15 min)

Open [`rust/lance-encoding/src/encoder.rs`](../rust/lance-encoding/src/encoder.rs). Read lines 1–200.

Core types:

- **`EncodedPage`** — a chunk of encoded bytes for one column, with metadata: rows, top-level row number, encoding description.
- **`EncodedColumn`** — the per-column rollup: the list of pages plus column-level buffers.
- **`EncodedBatch`** — a complete write unit (multiple columns).
- **`FieldEncoder`** — trait. One per Arrow field. Builds up state as `RecordBatch`es flow in, emits `EncodedPage`s when ready.
- **`FieldEncodingStrategy`** — chooses a `FieldEncoder` for each field given its type, options, and target version. Lance ships `StructuralEncodingStrategy` for 2.x.
- **`BatchEncoder`** — wraps a `FieldEncoder` per column and orchestrates encoding a whole `RecordBatch`.

Compression configuration flows through `CompressionParams` and `CompressionStrategy` (default is `DefaultCompressionStrategy`, defined in `compression.rs` and `compression_config.rs`).

Don't read every line — about 1100 lines. Read the trait declarations, the `EncodingOptions` struct, and the `default_encoding_strategy()` function.

### 1.5 The decoder side, structurally (10 min)

Re-open [`rust/lance-encoding/src/decoder.rs`](../rust/lance-encoding/src/decoder.rs). Skip to the type declarations.

Look for these traits (use editor outline or `rg "pub trait"`):

- `FieldScheduler` — knows how to figure out, for one logical field, which pages to schedule and in what order.
- `PageScheduler` — translates a single page request into I/O ranges and yields a decoder.
- `LogicalPageDecoder` — multi-column → Arrow array.
- `PhysicalPageDecoder` — single column → raw values.

The orchestrator is `schedule_and_decode` (search for it in `decoder.rs`). It accepts an `EncodingsIo`, a column layout, and a row range, and yields a `Stream<Item = ReadBatchTask>` where each task decodes some rows into a `RecordBatch`.

You don't need to read all 122 KB of `decoder.rs`. You need to know **the shape**: page-level scheduling, page-level decoding, both wired together by `schedule_and_decode`.

### 1.6 Repetition and definition (repdef) levels (15 min)

Open [`rust/lance-encoding/src/repdef.rs`](../rust/lance-encoding/src/repdef.rs) and read the **module-level doc comment** (lines 1–~150). It has worked examples.

The key idea, recycled from the Dremel paper / Parquet:

> Nested (list / struct) Arrow arrays have separate offset and validity buffers per layer. Storing them all separately on disk is wasteful when a missing outer element also implies missing inner elements. Repetition levels and definition levels collapse this redundancy into two dense integer arrays.

You don't need to derive repdef levels by hand. You need to recognize:

- `RepDefBuilder` — used by encoders to accumulate repdef as data streams in.
- `DefinitionInterpretation` — small enum that says "how many nullable layers were there?"
- The repdef arrays are written **alongside** the values as part of structural encodings.

Lance's repdef differs from Parquet's (different conventions about which level means "outer"). Don't mix them up if you've seen Parquet code.

### 1.7 Logical encodings (5 min)

`ls rust/lance-encoding/src/encodings/logical/`:

- `primitive.rs` — by far the biggest (~280 KB). All scalar types (ints, floats, decimals, bools, strings, binary, dictionary).
- `list.rs` — variable-length lists.
- `fixed_size_list.rs` — fixed-size lists (used heavily by vector columns).
- `struct.rs` — struct columns.
- `map.rs` — map columns.
- `blob.rs` — blob/blob-v2 (large value storage).

Each file has both an encoder and a logical decoder (or "structural encoder" / "field scheduler" pair).

Don't read these end-to-end. Skim `primitive.rs` lines 1–200 to see the interface and the dispatch on data type.

### 1.8 Physical encodings (5 min)

`ls rust/lance-encoding/src/encodings/physical/`:

- `value.rs` — the simple raw-value encoding.
- `bitpacking.rs` — bit-packed integers.
- `rle.rs` — run-length encoded.
- `binary.rs` — variable-length binary (with offsets).
- `fsst.rs` — Fast Static Symbol Table compression for strings.
- `block.rs` — generic compression block (zstd, lz4, etc.).
- `byte_stream_split.rs` — for floating-point compression.
- `packed.rs` — packed struct encoding.
- `general.rs` — generic compression dispatcher.
- `constant.rs` — constant column.

Again, don't read all of these. Open `value.rs` and `bitpacking.rs` lines 1–100 to see the simplest and the more interesting cases.

**End of Block 1.** Take a 10-min break.

## Block 2 — The on-disk format protos (60 min)

The encoding tree on disk is described entirely in protobufs. Reading the protos is the fastest way to grasp "what's actually written."

### 2.1 Encodings v2.1 — `protos/encodings_v2_1.proto` (30 min)

Open [`protos/encodings_v2_1.proto`](../protos/encodings_v2_1.proto). It's ~22 KB — read it in full. The comments are exceptionally thorough.

Pay attention to:

- **The "counting terms" section** at the top. `num_rows`, `num_items`, `num_visible_items`, `num_levels`, `num_buffers`. Don't gloss over these — they recur across the whole encoding system.
- **Structural encodings** — `MiniBlockLayout`, `FullZipLayout`. The miniblock layout is for "fat" pages with many small encoded blocks for random access. The full zip layout is for "thin" pages with values, validity, and repdef interleaved.
- **Compressive encodings** — `Variable`, `Constant`, `Bitpacked2`, `Fsst2`, `BlockCompression`. These describe how a single buffer of values is compressed.
- **The wrapper messages** — `PageLayout`, `ColumnInfo`, `PageInfo`. These are what the file index actually points at.

This proto is the single most important document for understanding the file format. **Read it twice.** Mark the structural-vs-compressive distinction in your head.

### 2.2 File proto — `protos/file2.proto` (15 min)

Open [`protos/file2.proto`](../protos/file2.proto) and read the module comment at the top:

```
* Each Lance file contains between 0 and 4Gi columns
* Each column contains between 0 and 4Gi pages
* Each page contains between 0 and 2^64 items
* Different pages within a column can have different items counts
* Columns may have up to 2^64 items
* Different columns within a file can have different item counts

The Lance file format does not have any notion of a type system or schemas.
From the perspective of the file format all data is arbitrary buffers of
bytes with an extensible metadata block to describe the data.
```

That last paragraph is a big idea: **the file format itself has no type system.** The schema lives in `lance-table`'s manifest, not in the file. A `.lance` file is just buffers + metadata. The encoding protos in 2.1 describe *what those buffers mean*. This separation lets the file format evolve independently of the schema/type system.

Read the rest of `file2.proto` for the messages: `Footer`, `ColumnMetadata`, `BufferDescriptor`, etc. The footer references positions within the file; the encoding proto tells you how to interpret each buffer.

### 2.3 Format module — `lance-encoding/src/format.rs` (15 min)

Open [`rust/lance-encoding/src/format.rs`](../rust/lance-encoding/src/format.rs). Read top to bottom (~30 KB but a lot is tests).

This is the Rust side of the protos. It exposes the proto types via the `pb` and `pb21` modules and adds Rust-side helpers in `ProtobufUtils`. When you build any `ArrayEncoding` proto in code, you go through `ProtobufUtils`.

Look at the helpers that build common encoding trees: `flat`, `bitpacked`, `nullable`, `dictionary`, `binary`, `fsst`, `packed`. These are the constructors used by the encoder strategies.

**End of Block 2.** Stretch.

## Block 3 — `lance-file`: writing and reading a file (60 min)

`lance-file` is the stitching layer between `lance-encoding` (which knows about pages and encodings) and the rest of the world (`lance-table`, the dataset layer).

### 3.1 Crate overview — `lance-file/src/lib.rs` (5 min)

Read it fully:

```rust
pub mod datatypes;
pub mod format;
pub(crate) mod io;
pub mod previous;
pub mod reader;
pub mod testing;
pub mod writer;

pub use io::LanceEncodingsIo;
```

The headline function is `determine_file_version`:

```rust
pub async fn determine_file_version(store: &ObjectStore, path: &Path, known_size: Option<usize>)
    -> Result<LanceFileVersion>
```

Reads the last 8 bytes of a file (`major | minor | "LANC"` magic), validates, returns the version. Every file open starts here.

### 3.2 The file format on disk (5 min)

Open [`rust/lance-file/src/format.rs`](../rust/lance-file/src/format.rs). It's tiny (~30 lines) and re-exports proto types and constants. Note `MAGIC = "LANC"`.

Open [`protos/file.proto`](../protos/file.proto) — this is the v1 file proto. Glance at it for context but focus on the v2 protos.

**ASCII layout reminder** (this also lives in `protos/table.proto` near the top):

```
+----------------------------------------+
|       Encoded Column 0, Chunk 0        |
|                  ...                   |
|       Encoded Column M, Chunk N        |
|       Indices                          |
|       Chunk Position (M x N x 8)       |
|         Manifest (Optional)            |
|         Metadata                       |
| i64: metadata position                 |
| MAJOR_VERSION | MINOR_VERSION | "LANC" |
+----------------------------------------+
```

The reader strategy: **always start at the end** (read the magic + metadata position), then jump to the metadata and walk forward.

### 3.3 FileWriter — `lance-file/src/writer.rs` (25 min)

Open [`rust/lance-file/src/writer.rs`](../rust/lance-file/src/writer.rs). It's 67 KB; here's how to navigate.

1. Lines 1–60: imports, constants. Note `PAGE_BUFFER_ALIGNMENT = 64` — pages are 64-byte aligned to play nicely with SIMD reads.
2. Find `pub struct FileWriter` and read its fields. It owns:
   - An async `Writer` (sink for bytes)
   - A `BatchEncoder` (from `lance-encoding`) that turns `RecordBatch`es into `EncodedPage`s
   - Accumulated state: column metadata, page positions, column buffers
3. Find `FileWriter::try_new` — the constructor.
4. Find `FileWriter::write_batch` (or similar) — the per-batch entry. It:
   - Pushes the batch into the encoder
   - Pulls out any `EncodedPage`s the encoder is ready to emit
   - Writes their buffers, padding to alignment
   - Updates internal page position bookkeeping
5. Find `FileWriter::finish` — closes the file:
   - Flushes any remaining encoder state
   - Writes column metadata
   - Writes the file footer (metadata position, magic)
6. Skim everything else.

Things to notice:

- **The writer never holds the whole file in memory.** Pages are flushed as soon as they're full enough.
- **Buffer alignment is enforced manually** with `PAD_BUFFER`.
- **`MAX_PAGE_BYTES = 32 MB`** is a writer-side cap (in 2.0, large pages cause downstream problems; in 2.1 the cap is removed because pages can be split on read).

### 3.4 FileReader — `lance-file/src/reader.rs` (20 min)

Open [`rust/lance-file/src/reader.rs`](../rust/lance-file/src/reader.rs). It's 90 KB — same strategy.

1. Lines 1–60: imports. Note the import of `schedule_and_decode` from `lance_encoding`.
2. Find `FileReaderOptions` — options for the reader (cache size, decoder plugins, etc.).
3. Find `pub struct FileReader` and read its fields.
4. Find `FileReader::try_open` — the constructor:
   - Determines the file version
   - Reads the footer (last bytes)
   - Reads the metadata at the position pointed to by the footer
   - Decodes the column metadata protos
   - Builds the in-memory `ColumnInfo`/`PageInfo` tables
5. Find `FileReader::read_stream` (or similarly named) — the read entry point:
   - Builds a `SchedulerDecoderConfig`
   - Calls `schedule_and_decode`, passing a `LanceEncodingsIo` (the I/O adapter)
   - Returns a `Stream<Item = Result<RecordBatch>>`

The bridge between `FileReader` and the scheduler is `LanceEncodingsIo` — defined in `lance-file/src/io.rs`. It implements `EncodingsIo` by delegating to a `FileScheduler` from `lance-io`. **This is the seam** between the I/O layer and the encoding layer.

### 3.5 Legacy reader — `lance-file/src/previous/` (5 min)

`previous/reader.rs`, `previous/writer/`, `previous/format/`, `previous/page_table.rs` implement the v1 file format. They exist purely for backwards compatibility. Skim the directory listing; don't read the contents unless a test fails on an old file.

**End of Block 3.** Stretch.

## Block 4 — Hands-on: see real bytes (45 min)

### Step 1: Write a tiny dataset (5 min)

Adapt the `write_read_ds` example to write *just one row* of a single integer column:

```rust
// In a scratch binary, e.g., rust/examples/src/bin/scratch.rs
use std::sync::Arc;
use arrow_array::{Int32Array, RecordBatch, RecordBatchIterator};
use arrow_schema::{DataType, Field, Schema};
use lance::dataset::WriteParams;
use lance::Dataset;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let schema = Arc::new(Schema::new(vec![Field::new("x", DataType::Int32, false)]));
    let batch = RecordBatch::try_new(schema.clone(),
        vec![Arc::new(Int32Array::from(vec![42_i32]))])?;
    let reader = RecordBatchIterator::new([Ok(batch)], schema);
    Dataset::write(reader, "/tmp/scratch.lance",
        Some(WriteParams::default())).await?;
    Ok(())
}
```

Or just edit `write_read_ds.rs` temporarily to point at `/tmp/scratch.lance` and a single small batch.

Run it: `cargo run -p lance-examples --example write_read_ds`.

### Step 2: Inspect the directory structure (5 min)

```bash
find /tmp/scratch.lance -type f | sort
```

You'll see something like:

```
/tmp/scratch.lance/_versions/1.manifest
/tmp/scratch.lance/_transactions/0-...txn
/tmp/scratch.lance/data/<uuid>.lance
```

Three things:

- `data/*.lance` — the actual data files in the format you just learned about.
- `_versions/N.manifest` — the manifest for version N. (More on this tomorrow.)
- `_transactions/*.txn` — the transaction record. (Also tomorrow.)

### Step 3: Look at the file footer (10 min)

The last 8 bytes of any `data/*.lance` file are `major(2) | minor(2) | "LANC"(4)`:

```bash
DATA_FILE=$(find /tmp/scratch.lance/data -name '*.lance' | head -1)
xxd "$DATA_FILE" | tail -3
```

You should see something like:

```
00000XXX: ...  ... .. .. .. .. 4c 41 4e 43  ......LANC
```

(the `4c 41 4e 43` is `LANC` in ASCII).

The 8 bytes before the magic are an `i64` (little-endian) pointing at the `Footer`/metadata. To read it without writing code:

```bash
# read the i64 at offset (file_size - 16)
SIZE=$(stat -f%z "$DATA_FILE")  # macOS; use stat -c%s on Linux
OFFSET=$((SIZE - 16))
dd if="$DATA_FILE" bs=1 skip=$OFFSET count=8 2>/dev/null | xxd
```

The bytes you see are a little-endian `u64` — the offset where the column metadata begins.

### Step 4: Use the lq CLI to inspect (5 min)

Lance has a built-in inspection CLI:

```bash
cargo run -p lance --bin lq -- --help
```

Try the subcommands. `lq` is in [`rust/lance/src/bin/lq.rs`](../rust/lance/src/bin/lq.rs). It's a great example of practical usage of the public API.

### Step 5: Read with `lance-tools` (10 min)

`lance-tools` has more inspection utilities. Browse `rust/lance-tools/src/` to see what's available, and run any binaries the CLI exposes.

### Step 6: Trace through `FileReader::try_open` (10 min)

With `/tmp/scratch.lance/data/<uuid>.lance` in mind, mentally trace:

1. `Dataset::open("/tmp/scratch.lance")` → reads the manifest (day 4).
2. The manifest names a fragment that names the data file path.
3. The dataset opens the data file via `FileReader::try_open(scheduler, path, ...)`.
4. `FileReader` reads the last 8 bytes: validates `LANC` and the version.
5. Reads the next-to-last 8 bytes: the metadata offset.
6. Reads the metadata range and decodes the protos.
7. Now the reader knows `ColumnInfo`/`PageInfo` for every column.
8. To read row 0, it asks: "for column 0, which page contains row 0?" Looks up the `PageInfo`, gets the byte range and encoding, schedules the read, and decodes.

You will not read all of these in code today — but you should be able to find each step's implementation when needed.
