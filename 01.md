## 1. Async, `.await`, `tokio`

LanceDB is **async-first**. Almost every public method returns a `Future`
(written as `async fn` or `-> impl Future`). You drive these futures with the
**Tokio** runtime.

Look at `rust/lancedb/examples/simple.rs`:

```rust
#[tokio::main]
async fn main() -> Result<()> {
    let db = connect("data/sample-lancedb").execute().await?;
    println!("{:?}", db.table_names().execute().await?);
    ...
}
```

What's happening:

* `#[tokio::main]` is a macro that wraps `main` so Tokio can drive your
  futures. Without an executor, `async` functions never run.
* `async fn foo() -> T` desugars to `fn foo() -> impl Future<Output = T>`.
* `.await` says "pause this task until the future resolves; meanwhile let
  other tasks run on the same thread".
* `connect(...)` returns a *builder*, not a future. `.execute()` consumes the
  builder and returns a future. `.await` resolves the future. `?` propagates
  the error.

You don't need to understand executors / pinning / wakers to read LanceDB
code. You **do** need to:

* Recognize `async fn` and that `.await` is required to actually run it.
* Know that `#[tokio::main]` only goes on the binary entry point — library
  code never calls it.

In doc-tests (e.g. `lib.rs`), you'll see this pattern:
```rust
# tokio::runtime::Runtime::new().unwrap().block_on(async {
let db = lancedb::connect("data/sample-lancedb").execute().await.unwrap();
# });
```
That's a manual runtime to drive a single future from a synchronous context.
It's only used inside doc examples.

**Read now**: `rust/lancedb/examples/simple.rs` (top to bottom, ~100 lines).

## 2. `Result<T>` and the `?` operator

LanceDB defines its own `Result`:

```rust
// rust/lancedb/src/error.rs
pub type Result<T> = std::result::Result<T, Error>;
```

The `Error` enum is built with **`snafu`**, a crate that gives you nice
`Display` impls, structured fields, and `From` conversions in one
declaration:

```rust
#[derive(Debug, Snafu)]
#[snafu(visibility(pub(crate)))]
pub enum Error {
    #[snafu(display("Table '{name}' was not found"))]
    TableNotFound { name: String, source: BoxError },
    ...
}
```

Two things to internalize:

* The **`?`** operator on a `Result<T, E1>` works in a function that returns
  `Result<U, E2>` only if there is a `From<E1> for E2` impl. Look at
  `rust/lancedb/src/error.rs` after the enum — there are many `From` impls
  (`From<ArrowError>`, `From<DataFusionError>`, `From<lance::Error>`,
  `From<object_store::Error>`, …). That's why you can write
  `arrow_op()?` inside a function returning `lancedb::Result<T>`.
* Several variants wrap a `BoxError = Box<dyn std::error::Error + Send +
  Sync>`. That's the type-erased "any error" container; LanceDB uses it for
  errors that originate in user code or in dependencies it doesn't want to
  hardcode against.

`#[snafu(transparent)]` on the `External` variant and `#[snafu(whatever, ...)]`
on the `Other` variant are escape hatches: "transparent" means "just delegate
`Display` / `source` to the inner error", and "whatever" means "this is the
catch-all variant the `whatever_context!` macros build into".

**Read now**: all of `rust/lancedb/src/error.rs` (~200 lines). Don't memorize
the variants, just notice the *shape*: each variant carries the data you'd want
in a log line, and `From` impls make `?` ergonomic.

## 3. `Arc<T>` and `Arc<dyn Trait>`

`Arc<T>` is "atomic reference-counted shared ownership". You'll see it
everywhere because:

* LanceDB returns objects that you want to *share* (a `Connection`, a `Table`)
  but doesn't want you to manage lifetimes for.
* Async code crosses thread boundaries (Tokio is multi-threaded by default), so
  things shared across `.await` points need `Send`. `Arc<T>` is `Send` when
  `T` is `Send`.

`Arc<dyn Trait>` is the typical way to do **dynamic dispatch with shared
ownership**. Examples:

```rust
// rust/lancedb/src/connection.rs
parent: Arc<dyn Database>,

// rust/lancedb/src/database.rs
async fn open_table(&self, request: OpenTableRequest) -> Result<Arc<dyn BaseTable>>;

// rust/lancedb/src/embeddings.rs (registry)
Arc<dyn EmbeddingFunction>
```

Why dynamic dispatch? Because the `Database` could be a local listing one, a
remote HTTP one, or a namespace-managed one — the calling code shouldn't
care, it just calls trait methods.

`Arc::clone(&x)` (or just `x.clone()`) is **cheap**: it bumps a refcount, no
deep copy. So you'll see `Arc::clone` liberally in builders, and that's fine.

**Mental model**: when you see `Arc<dyn Trait>`, read it as "shared handle to
some object that implements `Trait`; cheap to clone; can be sent between
threads".

## 4. The Builder Pattern with `mut self`

This is *the* dominant pattern in LanceDB's public API. The shape is:

```rust
let db = connect("data/sample-lancedb")
    .storage_options([("aws_access_key_id", "key"), ("aws_secret_access_key", "secret")])
    .execute()
    .await?;
```

`connect(...)` returns a `ConnectBuilder`. Each method like
`storage_options(...)` takes `mut self` (note: not `&mut self`!) and returns
`Self`. That moves the builder, mutates it, returns it. Then `.execute()`
consumes the builder by value and produces the future.

Look at `OpenTableBuilder` in `rust/lancedb/src/connection.rs`:

```rust
pub fn index_cache_size(mut self, index_cache_size: u32) -> Self {
    self.request.index_cache_size = Some(index_cache_size);
    self
}
```

Why builders instead of long argument lists?

* Optional parameters with defaults — only set what you need.
* **Forward-compatible**: you can add a new optional method without breaking
  callers. (See "Rust API design" in `AGENTS.md`.)
* Fluent style reads naturally.

You'll see this pattern for: `ConnectBuilder`, `OpenTableBuilder`,
`CreateTableBuilder`, `Query`, `VectorQuery`, `AddDataBuilder`,
`UpdateBuilder`, `MergeInsertBuilder`, `IvfPqIndexBuilder`, every index
builder, and many more.

**Tip**: when reading a builder, find the `execute()` (or sometimes the
terminal `.await`-able method) at the end. That's where the "real work"
happens. Everything before it is just configuration.

## 5. `async_trait`, `BaseTable`, and `Database`

In stable Rust, you can't yet write `async fn foo(&self) -> T` directly inside
a trait without some workarounds. The crate
[`async-trait`](https://docs.rs/async-trait/) papers over this with a macro:

```rust
#[async_trait]
pub trait BaseTable: ... {
    async fn schema(&self) -> Result<SchemaRef>;
    async fn count_rows(&self, filter: Option<Filter>) -> Result<usize>;
    ...
}
```

What it does behind the scenes: rewrites each `async fn` into
`fn ... -> Pin<Box<dyn Future<...> + Send + '_>>`. You don't need to think
about that — just know that any trait you see decorated with `#[async_trait]`
has async methods that you can `.await` like normal.

The two big traits driven by `async_trait` in LanceDB:

* `Database` (in `rust/lancedb/src/database.rs`): represents a *catalog* —
  something that knows how to list/create/open tables.
* `BaseTable` (in `rust/lancedb/src/table.rs`): represents an opened *table* —
  schema, count, scan, add, delete, update, create_index, etc.

Each has multiple impls:

* `Database`: `ListingDatabase`, `LanceNamespaceDatabase`, `RemoteDatabase`
  (`#[cfg(feature = "remote")]`).
* `BaseTable`: `NativeTable` (local Lance), `RemoteTable`.

This **trait-object pluggability** is how the same `lancedb::Connection` and
`lancedb::Table` work for local and cloud.

## 6. `Into<String>` and `AsRef<T>` in arguments

The `AGENTS.md` style guide explicitly recommends this. You'll see signatures
like:

```rust
pub fn storage_option(mut self, key: impl Into<String>, value: impl Into<String>) -> Self
```

instead of `key: String, value: String`. Why: callers can pass `&str`,
`String`, `Cow<'_, str>`, `Box<str>`, anything that implements `Into<String>`,
without typing `.to_string()` at every call site. It's purely an ergonomic
choice.

`AsRef<T>` is similar but doesn't take ownership — used when the callee only
needs a reference.

When *writing* code: prefer `impl Into<String>` for owned-string params and
`impl AsRef<str>` for borrowed ones, in public APIs.

## 7. Errors from third-party crates: `BoxError` and `From` impls

The `External` variant of `Error` (in `error.rs`) is `#[snafu(transparent)]`.
Combined with the `from_box_error` helper, this lets LanceDB:

* Accept any `Box<dyn std::error::Error + Send + Sync>` and either downcast it
  to a known LanceDB error, or wrap it.
* Avoid an explosion of "lance::Error::Wrapped(arrow::Error::ExternalError(...))"
  layers in your error message — `from_box_error` peels those onion layers off.

You don't need to understand `from_box_error` deeply. Just know:

* Functions in user code that fail can return `Box<dyn Error + Send + Sync>`
  and LanceDB will accept that and convert it.
* When you see weird error chains in tests, look at this function — it's
  probably involved.

## 8. `Send + Sync + 'static`

These bounds appear all over trait definitions:

```rust
pub trait Database: Send + Sync + std::any::Any + std::fmt::Debug + ... + 'static
```

* `Send` = "ownership of `T` can be transferred to another thread".
* `Sync` = "`&T` can be shared across threads".
* `'static` = "this type doesn't borrow anything with a shorter lifetime than
  the program". (For trait objects, `dyn Trait + 'static` is the default if
  unspecified.)

Tokio's executor moves tasks between threads, so any value held across an
`.await` must be `Send`. Anything stored in an `Arc<dyn Trait>` that is shared
across threads needs `Sync`.

When you see `Send + Sync + 'static`, read it as "this is a trait object the
Tokio runtime can move around freely".

## 9. Streams: `futures::Stream` and `RecordBatchStream`

A `Stream<Item = T>` is the async equivalent of `Iterator<Item = T>`. LanceDB
returns query results as a *stream of `RecordBatch`es* rather than a single
buffer, because results can be huge.

In `rust/lancedb/src/arrow.rs`:

```rust
pub trait RecordBatchStream: Stream<Item = Result<arrow_array::RecordBatch>> {
    fn schema(&self) -> Arc<arrow_schema::Schema>;
}

pub type SendableRecordBatchStream = Pin<Box<dyn RecordBatchStream + Send>>;
```

You consume a stream with combinators from the `futures` crate:

```rust
use futures::TryStreamExt;
let batches = table.query().execute().await?
    .try_collect::<Vec<_>>().await?;
```

* `try_collect` collects all `Ok(item)` into a `Vec`, propagating the first
  `Err`.
* `try_next().await` pulls one batch.
* `.map_err(...)`, `.try_filter(...)`, `.boxed()` — all common.

You'll meet streams in Day 2 (when scanning a table) and Day 3 (when reading
query results). Don't try to pre-learn them; just know they're "iterators that
require `.await`".

## 10. Cargo features

Open `rust/lancedb/Cargo.toml`. Look at the `[features]` block:

```toml
[features]
default = []
aws = [ "lance/aws", "lance-io/aws", "object_store/aws", ... ]
remote = ["dep:reqwest", "dep:http", ... ]
openai = ["dep:async-openai", "dep:reqwest"]
sentence-transformers = [...]
```

A **cargo feature** is a compile-time flag that turns on optional dependencies
or code paths. In source you'll see:

```rust
#[cfg(feature = "remote")]
pub mod remote;
```

That module only exists if you compile with `--features remote`. The repo's
canonical command (`cargo check --quiet --features remote --tests --examples`)
turns on the most common one. If you ever see "cannot find module `remote`"
while reading, it's because your IDE didn't enable the feature — fix this in
your rust-analyzer settings.
