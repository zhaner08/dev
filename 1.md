## Block 1 — Rust concepts (90 min)

I will not re-teach what the Rust Book teaches well. Instead, this block is the **delta** between "I know `let mut x = 5;`" and "I can read `lance/src/dataset.rs` line by line." Every concept is paired with a real snippet from the codebase. Open the file alongside.

### 1.1 Ownership & borrowing — the 30-second refresh

- A value has exactly one owner.
- You can lend it out as `&T` (read-only, many at a time) or `&mut T` (read-write, one at a time).
- When the owner goes out of scope, the value is dropped.

This sounds minor; in practice it's why Rust APIs look the way they do.

**Lance example.** Open [`rust/lance/src/lib.rs`](../rust/lance/src/lib.rs) and look at the doctest near the top:

```rust
let dataset = Dataset::open(&path).await.unwrap();
let mut scanner = dataset.scan();
let batches: Vec<RecordBatch> = scanner
    .try_into_stream()
    .await
    .unwrap()
    .map(|b| b.unwrap())
    .collect::<Vec<RecordBatch>>()
    .await;
```

Why is `scanner` `mut`? Because building a scan plan mutates the builder (`scanner.filter(...)`, `.limit(...)`). Why is `dataset` not `mut`? Because `Dataset` is logically read-only — every "mutation" produces a new version (this is core to Lance's design).

### 1.2 `String` vs `&str`, `Vec<T>` vs `&[T]`, `Path` vs `&Path`

You will see all four constantly. Rule of thumb:

- **Function parameters** prefer the borrowed form (`&str`, `&[T]`, `&Path`) so callers don't have to clone.
- **Function return values** typically use the owned form when the function had to allocate.
- **Struct fields** typically use the owned form (`String`, `Vec<T>`, `PathBuf`) because the struct outlives any caller.

You'll occasionally see `Cow<'a, str>` (Clone-on-write) and `Arc<str>` for "I want cheap clones." Lance uses `Arc<str>` and `Arc<String>` in cache keys.

### 1.3 `Option<T>` and `Result<T, E>` — and the `?` operator

You know these. Things to internalize:

- `Result<T, E>` chained via `?` is **the** error-handling pattern. Lance has its own `lance_core::Result<T> = Result<T, lance_core::Error>` alias used everywhere.
- `Option<T>::ok_or_else` turns `None` into a custom error. Used to convert "missing" into a typed `Error::NotFound`.
- `Option<T>::is_some` / `Option<T>::is_none` followed by `.unwrap()` is an anti-pattern in this codebase (see `rust/AGENTS.md`). Prefer `if let Some(x) = ... { ... }`.

**Lance example.** Open [`rust/lance-core/src/error.rs`](../rust/lance-core/src/error.rs) lines ~70–230 to see the full `Error` enum. Notice:

```rust
#[derive(Debug, Snafu)]
#[snafu(visibility(pub))]
pub enum Error {
    #[snafu(display("Invalid user input: {source}, {location}"))]
    InvalidInput {
        source: BoxedError,
        #[snafu(implicit)]
        location: Location,
    },
    // ...
}
```

The `Snafu` derive macro auto-generates a "context selector" struct (`InvalidInputSnafu`) you use like:

```rust
return Err(Error::invalid_input("schema mismatch"));
// or, when wrapping another error:
some_op().context(InvalidInputSnafu)?;
```

The `Location` field is filled in **automatically at the call site** thanks to `#[snafu(implicit)]`. You'll see this everywhere: every Lance error carries its own `file:line` for free.

### 1.4 Smart pointers: `Box`, `Arc`, `Rc`, `Cell`/`RefCell`, `OnceLock`/`LazyLock`

You probably know `Box<T>` (heap allocation). The others, in order of how often you'll meet them in Lance:

- **`Arc<T>`** — atomically reference-counted. The single most common smart pointer in this codebase. Lance shares schemas, manifests, object stores, and caches across many threads; each is wrapped in `Arc`. Cloning an `Arc` is cheap (just bumps a counter); cloning the inner `T` is expensive.
- **`LazyLock<T>`** — global lazily-initialized constants. Replaces the older `lazy_static!` macro. Example from `lance-core/src/lib.rs`:

  ```rust
  pub static ROW_ID_FIELD: LazyLock<ArrowField> =
      LazyLock::new(|| ArrowField::new(ROW_ID, DataType::UInt64, true));
  ```
- **`OnceLock<T>`** — same idea but for non-`const`-initializable values that are set once at runtime.
- **`Rc<T>`** — like `Arc` but not thread-safe. Rare in Lance because almost everything crosses thread boundaries.
- **`Cell<T>` / `RefCell<T>`** — interior mutability without locks. Also rare; we prefer `parking_lot::Mutex` or `tokio::sync::Mutex` when needed.

**Why so many `Arc`s?** Because Rust's "one owner" rule is too strict for an async, multi-thread world. `Arc<dyn Trait>` is how you say "this thing is shared, polymorphic, and lives as long as anyone holds a reference."

Open [`rust/lance/src/dataset.rs`](../rust/lance/src/dataset.rs) lines ~135–170 and look at the `Dataset` struct. Count the `Arc`s — there are seven. Each one is a piece of state that's expensive to clone but needs to be shared.

### 1.5 Traits and generics — the heart of Rust

You will see four trait patterns daily.

**Pattern A: associated types.** A trait that has a "this implementor's choice" type.

```rust
pub trait EncodingsIo: std::fmt::Debug + Send + Sync {
    fn submit_request(
        &self,
        range: Vec<Range<u64>>,
        priority: u64,
    ) -> BoxFuture<'static, Result<Vec<Bytes>>>;
}
```

(From [`rust/lance-encoding/src/lib.rs`](../rust/lance-encoding/src/lib.rs).) The `: std::fmt::Debug + Send + Sync` part is **trait bounds** — it requires implementors to also be debuggable and thread-safe.

**Pattern B: trait objects (`dyn Trait`).** When you need polymorphism at runtime: "I have a vector of *some* `Index` implementations, but I don't know which type each one is." You'll see `Arc<dyn Index>` and `Box<dyn Future<...>>` constantly.

**Pattern C: generic functions with bounds.**

```rust
pub async fn open_dataset<T: AsRef<str>>(table_uri: T) -> Result<Dataset> { ... }
```

This says "I take any type `T` that can be referenced as a `&str`" — works for `String`, `&str`, `&String`, `Path` (sort of), etc. `AsRef`/`Into`/`From` are the workhorse traits for "be flexible about input types."

**Pattern D: `async-trait`.** Rust didn't natively support `async fn` in traits until late 2023, and even now it has restrictions. You'll see this annotation everywhere:

```rust
#[async_trait]
pub trait Index: Send + Sync + DeepSizeOf {
    async fn prewarm(&self) -> Result<()>;
    // ...
}
```

The `#[async_trait]` macro rewrites `async fn` to `fn ... -> Pin<Box<dyn Future<...>>>` under the hood. You can read it as "async fn in a trait."

### 1.6 Async/await — the 60-second model

**Mental model.** An `async fn` returns a `Future<Output = T>` instead of `T`. A `Future` is a state machine that, when polled, either makes progress and returns `Poll::Ready(value)` or returns `Poll::Pending`. A **runtime** (in our case `tokio`) drives the polling and schedules tasks across threads.

Concretely:

```rust
let dataset = Dataset::open(&uri).await?;
```

The `.await` says "suspend this function until the future is `Ready`, then continue." While suspended, tokio runs other work. This is why Lance can fan out hundreds of parallel I/O requests on a few OS threads.

**Things to actually remember:**

- `async fn foo() -> T` is shorthand for `fn foo() -> impl Future<Output = T>`.
- You can only `.await` inside an `async` function or block.
- Top-level entry is `#[tokio::main] async fn main()` or `tokio::runtime::Runtime::new()`.
- A `Stream<Item = T>` is the async version of `Iterator<Item = T>`. Use `.next().await` or combinators from `futures::StreamExt`.

**Real example — a Lance scan.** Open [`rust/examples/src/write_read_ds.rs`](../rust/examples/src/write_read_ds.rs):

```rust
async fn read_dataset(data_path: &str) -> Result<(), Box<dyn std::error::Error>> {
    let dataset = Dataset::open(data_path).await?;
    let scanner = dataset.scan();
    let mut batch_stream = scanner.try_into_stream().await?.map(|b| b.unwrap());
    while let Some(batch) = batch_stream.next().await {
        println!("Batch size: {}, {}", batch.num_rows(), batch.num_columns());
    }
    Ok(())
}
```

Three `.await`s: opening the dataset (reads the manifest), starting the stream (sets up the I/O pipeline), and pulling each batch. Internally there are thousands of futures, but at this level the code reads almost like sync code.

### 1.7 Lifetimes — when you'll meet them, when to ignore them

Lifetimes are annotations like `'a` that tell the borrow checker how long a reference is valid. **Most Lance code does not have explicit lifetimes** — they're inferred. You'll see `'static` (lives forever) often, e.g. `BoxFuture<'static, Result<...>>`, which means "this future does not borrow any local variable."

When you do see `<'a>` parameters, it's usually in iterator/stream adapters and serialization code. For day 1, the rule is: **if you see a lifetime, skim past it.** By day 4 you'll start needing them.

### 1.8 Macros — the ones you must recognize

Rust has both declarative macros (`macro_rules!`) and procedural macros. You don't need to write them, but you must recognize them.

| Macro | What it does | Where it shows up |
|------|------|------|
| `vec![1, 2, 3]` | builds a `Vec` | everywhere |
| `format!("...")` | builds a `String` like `printf` | everywhere |
| `assert!`, `assert_eq!`, `debug_assert!` | invariants and tests | everywhere |
| `#[derive(...)]` | auto-implements traits | every struct |
| `#[tokio::test]` | async test entry point | tests |
| `#[rstest]`, `#[case::name(...)]` | parameterized tests | tests |
| `tracing::info!`, `instrument!` | structured logs and spans | hot paths |
| `include!(concat!(env!("OUT_DIR"), "/lance.pb.rs"))` | inline generated proto code | each `pb` module |

The last one is worth pausing on. Lance generates Rust code from `.proto` files at build time using `prost-build` (configured in each crate's `build.rs`). The output goes to `target/<profile>/build/<crate>-<hash>/out/lance.pb.rs` and is `include!`-ed where you see it. If you ever want to see the actual generated structs, run `cargo build` and then look in `target/`.

### 1.9 Modules, visibility, `pub use` re-exports

Rust's `mod` system maps to file/folder structure. Quick rules:

- `mod foo;` in `lib.rs` looks for `foo.rs` or `foo/mod.rs` in the same directory.
- `pub` exposes outside the crate; `pub(crate)` exposes within the crate; default is private to the module.
- `pub use ... as ...` re-exports items at a different path. Lance does this a lot to keep the public API surface clean.

Example: `lance::Dataset` is actually defined at `rust/lance/src/dataset.rs`, but `rust/lance/src/lib.rs` does `pub use dataset::Dataset;` so users can write `use lance::Dataset`.

### 1.10 What I'm deliberately skipping

Things you don't need today:

- `Pin<&mut T>` deep dive — `pin-project` makes this almost transparent in practice.
- Custom `Future` implementations — Lance has a few in `lance-io::scheduler`, but reading is enough.
- `unsafe` — there are pockets of it in encoding/SIMD; we'll point them out but you don't need to write any.
- Lifetime elision rules — the compiler will tell you.
- Trait coherence and the orphan rule — comes up only when you implement traits across crates.

## Block 2 — Cargo workspace mechanics (30 min)

Lance is a **Cargo workspace**: one `Cargo.toml` at the root declares many member crates. Skim [`Cargo.toml`](../Cargo.toml) at the repo root.

Key concepts:

### Workspace inheritance

The root `[workspace.package]` and `[workspace.dependencies]` sections are *inherited* by member crates that opt in. In `rust/lance/Cargo.toml` you'll see things like:

```toml
[package]
name = "lance"
version.workspace = true     # inherit from workspace
edition.workspace = true

[dependencies]
arrow-array.workspace = true # use the version pinned at workspace level
lance-core.workspace = true
```

This means **all Lance crates use the exact same version of every external dep**, which is critical for a workspace this large.

### The dependency graph (memorize this!)

At the workspace level, the Lance crates form a layered DAG:

```
lance (top-level)
 ├─ lance-table ──┐
 ├─ lance-index ──┤
 ├─ lance-datafusion ─┤
 │                    ├─ lance-file ──┐
 │                    │               ├─ lance-encoding ──┐
 │                    │               │                   ├─ lance-arrow ──┐
 │                    │               │                   │                ├─ lance-core
 │                    │               │                   ├─ lance-io  ────┤
 │                    │               └─ lance-io ────────┤
 ├─ lance-namespace ──┘                                   │
 └─ lance-tokenizer ─────────────────────────────────────────────────── lance-core
```

**Rules:**

- A crate may only depend on crates *below* it. `lance-core` depends on nothing else in the project.
- `lance-arrow` extends `arrow-rs` with Lance-specific helpers. It's a kitchen sink.
- The "physical" stack (encoding → file → table → dataset) stacks bottom-up.
- The "logical" stack (index, datafusion, namespace) is parallel and depends on the file/table layers.

### Feature flags

Several crates have Cargo features for optional behaviour:

- `lance/dynamodb` — DynamoDB external manifest store
- `lance-file/wasm` — WASM build (no `tokio::fs`, no uring)
- `lance-io/io-uring` — Linux-only fast async I/O
- `lance-index/protoc` — use the system `protoc` instead of the bundled one
- Geospatial features in `lance-geo`

You won't toggle these on day 1, but be aware they exist when something looks "missing."

### Building one crate vs the whole workspace

```bash
cargo check -p lance-core              # fastest: just one crate, no codegen
cargo check -p lance-encoding --tests  # check tests too
cargo check --workspace                # everything
cargo test -p lance-core               # run lance-core's tests
cargo doc --open -p lance              # build & open the top-level crate's docs in a browser
```

You'll live in `cargo check -p <crate>` while reading; it's much faster than full builds.

## Block 4 — Hands-on: build, run, read your first file (30 min)

### Step 1: Build the workspace (5 min if cached)

```bash
cargo check --workspace --tests --benches
```

If this is the first time, expect 5–15 minutes. Lance has lots of crates and codegen.

### Step 2: Run the canonical example

```bash
cargo run -p lance-examples --example write_read_ds
```

Expected output (truncated):

```
Batch size: 6, 2
Schema: Schema { fields: [...], ... }
Batch: RecordBatch { ... }
Cleaned up resources at: /var/folders/...
```

Read the example source: [`rust/examples/src/write_read_ds.rs`](../rust/examples/src/write_read_ds.rs). It's 80 lines and demonstrates:

- Building Arrow `RecordBatch`es
- `WriteParams` and `Dataset::write`
- `Dataset::open`, `dataset.scan()`, `try_into_stream`
- Async cleanup

You should be able to read every line of this file without confusion. If you can't, identify the line and ask me.

### Step 3: Read your first non-trivial source file

Open [`rust/lance/src/lib.rs`](../rust/lance/src/lib.rs) and read it fully. It's only 120 lines and contains:

- The two doctests at the top (write a dataset, scan a dataset)
- The crate's module declarations (`pub mod dataset;`, `pub mod index;`, etc.)
- The public API surface (`pub use dataset::Dataset;`)
- The `open_dataset` convenience function
- The `pb` module that brings in proto-generated code
- The `deps` re-export module (so users pin matching versions)

This is the "front door" to all of Lance. Notice how small it is — it does almost nothing itself; it just curates the public API and re-exports.

### Step 4: Pop the hood once

Open [`rust/lance/src/dataset.rs`](../rust/lance/src/dataset.rs) — but don't try to read it. Instead:

1. Read lines 1–80 (imports — get a sense of how many crates `Dataset` touches).
2. Jump to the `pub struct Dataset` declaration around line ~145 and read its fields. Each field's type is a clue: `Arc<ObjectStore>`, `Arc<Manifest>`, `Refs`, two caches.
3. Use your editor's outline view (`cargo doc --open -p lance` works too) to scroll the method list. There are dozens. Don't read them; just see the surface area.
4. Close the file. You're not ready for it yet — you will be by day 4.

That's the whole point of bottom-up: a 131 KB file becomes much smaller when you already know what `Manifest`, `ObjectStore`, and `Refs` are.
