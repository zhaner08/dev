## Block 1 — `lance-core` (75 min)

`lance-core` is small and full of building blocks. It has no Lance dependencies (only `arrow-rs`, `tokio`, etc.), so it's the right place to start.

### 1.1 The crate's lib.rs (5 min)

Open [`rust/lance-core/src/lib.rs`](../rust/lance-core/src/lib.rs). Read all 80 lines. Notice:

- `pub use error::{ArrowResult, Error, Result, box_error};` — these are the error types every other Lance crate re-uses.
- The `WILDCARD`, `ROW_ID`, `ROW_ADDR`, `ROW_OFFSET` constants — these are **column names** for system columns. When you see `_rowid` in a query, that's this constant.
- The `*_FIELD` `LazyLock`s are the corresponding Arrow `Field` objects.
- `is_system_column` — gate used in many places to skip or handle system columns specially.

This file is the public surface. Everything else in the crate is reachable through it.

### 1.2 Errors — `lance-core/src/error.rs` (15 min)

Open [`rust/lance-core/src/error.rs`](../rust/lance-core/src/error.rs). Read lines 1–230 (roughly through the end of the `enum Error` declaration). Skim the rest.

**What to learn:**

```rust
#[derive(Debug, Snafu)]
#[snafu(visibility(pub))]
pub enum Error {
    InvalidInput { source: BoxedError, location: Location },
    DatasetNotFound { path: String, source: BoxedError, location: Location },
    SchemaMismatch { difference: String, location: Location },
    NotFound { uri: String, location: Location },
    IO { source: BoxedError, location: Location },
    Index { message: String, location: Location },
    Internal { message: String, location: Location },
    NotSupported { source: BoxedError, location: Location },
    // ... and more
}
```

Things to internalize:

1. **Every variant has a `Location`.** Filled in by `snafu` automatically, no boilerplate at call sites. When you see a Lance error in production, the file:line where it was constructed is in the message.
2. **`source: BoxedError` is a type-erased inner error.** This is how Lance wraps errors from `arrow-rs`, `object_store`, `tokio`, etc., into a single Lance `Error`.
3. **There are constructor methods on `Error`** (look further down in the file): `Error::invalid_input(...)`, `Error::not_found(...)`, `Error::io(...)`. These are convenience wrappers around the snafu context selectors.
4. **`From` implementations** at the bottom of the file convert from `arrow_schema::ArrowError`, `object_store::Error`, `std::io::Error`, etc. This is what makes `?` work transparently across error types.
5. **`Result<T> = std::result::Result<T, Error>`** — the alias every Lance API uses.

Why this matters: when a future test or run fails with a Lance error, the location string tells you *where in lance source* the error was created. That's a powerful debugging shortcut.

**Mini-exercise:** find `Error::corrupt_file`. Where is it constructed in the codebase? Use `rg "Error::corrupt_file" rust/`.

### 1.3 Schema and Field — `lance-core/src/datatypes/` (20 min)

Lance has its own `Schema` and `Field` types that **wrap and extend** Arrow's. Why? Because Lance needs more metadata than Arrow ships with: per-field IDs (stable across writes), encoding hints, dictionary IDs, blob configuration, storage class, etc.

Open [`rust/lance-core/src/datatypes/field.rs`](../rust/lance-core/src/datatypes/field.rs) and read lines 1–200 to see the `Field` struct and its fundamental methods. Then skim. Highlights:

- A `Field` has a numeric `id` that **never changes** across schema evolution. Adding a new column gets a new id; renaming a column keeps the same id.
- `parent_id` makes this a tree (struct/list nesting).
- `extension_name` and `metadata` carry both Arrow and Lance-specific extras (e.g. blob marker).
- Dictionaries have their own `Dictionary` struct; field-id ↔ dictionary-id mapping lives on the field.

Open [`rust/lance-core/src/datatypes/schema.rs`](../rust/lance-core/src/datatypes/schema.rs) — at 105 KB, you do not read this end-to-end. Instead:

1. Read lines ~1–100 to see the imports, `Schema` struct, and constructors.
2. Search (`Ctrl-F`) for `pub fn project` — Lance has rich projection logic (column selection by name, by id, with options for missing/mismatched types).
3. Search for `pub fn merge` and `pub fn check_compatible` — schema evolution lives here.

**The big idea:** Arrow's `Schema` is a public-API container; Lance's `Schema` is the internal source of truth. There are conversions both ways (`From`/`TryFrom`).

### 1.4 RowAddress — `lance-core/src/utils/address.rs` (10 min)

Open the file in full. It's only ~120 lines and is a great example of "wrap a primitive in a newtype to add semantics."

```rust
pub struct RowAddress(u64);

impl RowAddress {
    pub const FRAGMENT_SIZE: u64 = 1 << 32;
    // upper 32 bits = fragment id, lower 32 bits = row offset in fragment
}
```

`RowAddress` ≠ `RowId`:

- **`RowAddress`** is a *physical* pointer: "fragment N, row offset M." Changes when fragments are compacted.
- **`RowId`** is a *stable logical identifier* that survives compaction. Lance maintains a row id index that maps `RowId` ↔ `RowAddress`.

`rust/AGENTS.md` has a strong rule: never mix the two. Always use `RowAddress` (the type), never raw `u64` arithmetic.

### 1.5 DeletionVector — `lance-core/src/utils/deletion.rs` (10 min)

Open and read lines 1–150 of [`rust/lance-core/src/utils/deletion.rs`](../rust/lance-core/src/utils/deletion.rs).

```rust
pub enum DeletionVector {
    NoDeletions,
    Set(HashSet<u32>),
    Bitmap(RoaringBitmap),
}
```

Three states for memory efficiency:

- `NoDeletions` — common case, costs zero bytes.
- `Set` — for small deletion counts (< 5,000 rows), `HashSet<u32>` is faster.
- `Bitmap` — for large counts, `RoaringBitmap` is more compact.

Auto-promotion from `Set` to `Bitmap` happens at the threshold. This pattern (small/large variants behind one enum) is common in Lance.

Each fragment has at most one deletion vector. The vector is persisted as a `.deletion` file inside the dataset.

### 1.6 Masks — `lance-core/src/utils/mask.rs` (10 min)

This is `RowAddrMask`, used by the prefilter and scanner to express "rows I want" or "rows I don't want" across multiple fragments at once.

Open [`rust/lance-core/src/utils/mask.rs`](../rust/lance-core/src/utils/mask.rs) and read lines 1–80.

```rust
pub enum RowAddrMask {
    AllowList(RowAddrTreeMap),
    BlockList(RowAddrTreeMap),
}
```

The `RowAddrTreeMap` is a `BTreeMap<u32, RoaringBitmap>` keyed by fragment id, holding row offsets per fragment. This is the canonical compact representation of "a set of `RowAddress`es."

Masks compose: filter A `AND` filter B becomes "intersect their allow-lists." A scalar index lookup returns one mask, a vector pre-filter returns another, and the scanner intersects them.

### 1.7 Cache — `lance-core/src/cache/` (5 min)

Skim [`rust/lance-core/src/cache/mod.rs`](../rust/lance-core/src/cache/mod.rs) lines 1–100. Key abstractions:

- `LanceCache` — typed, weight-aware concurrent cache, backed by the `moka` crate.
- `Capacity` — bytes-based capacity (Lance caches are bounded by RAM, not entry count).
- `WeakLanceCache` and key-prefixing — lets one underlying cache serve multiple datasets without leaking keys across them.

You'll see two caches on every `Dataset`:

- `metadata_cache` — manifests, schemas, transactions (~1 GiB by default)
- `index_cache` — index data structures (~6 GiB by default)

Both use this implementation.

### 1.8 Async helpers — `lance-core/src/utils/futures.rs` and friends (5 min)

Skim [`rust/lance-core/src/utils/futures.rs`](../rust/lance-core/src/utils/futures.rs) and [`rust/lance-core/src/utils/tokio.rs`](../rust/lance-core/src/utils/tokio.rs).

Things to recognize:

- `spawn_cpu` — spawns a CPU-intensive job on a dedicated thread pool (separate from the I/O runtime). Critical for not blocking I/O futures with k-means/PQ training.
- `get_num_compute_intensive_cpus` — caps CPU-bound parallelism.
- Various stream/future combinators for batching, backpressure, and error handling.

You'll see `lance_core::utils::tokio::spawn_cpu(...)` peppered through the encoding and indexing crates. It's where the compute heavy lifting happens.

### 1.9 Other useful bits

Files worth knowing exist (don't read deeply yet):

- `lance-core/src/utils/aimd.rs` — AIMD (Additive-Increase Multiplicative-Decrease) congestion-style backpressure. Used in the I/O scheduler.
- `lance-core/src/utils/backoff.rs` — exponential backoff for retries.
- `lance-core/src/utils/cpu.rs` — CPU feature detection (AVX2, AVX-512, NEON) for the SIMD paths.
- `lance-core/src/levenshtein.rs` — fuzzy "did you mean" suggestions for field-name errors.

**End of Block 1.** Take a 10-min break.

## Block 2 — `lance-arrow` (60 min)

`lance-arrow` is a "junk drawer" of arrow-rs extensions. Don't read it cover-to-cover; learn its layout so you can find things.

### 2.1 What's in the crate (5 min)

Open [`rust/lance-arrow/src/lib.rs`](../rust/lance-arrow/src/lib.rs) and look at the module declarations:

```rust
pub mod deepcopy;
pub mod schema;
pub mod bfloat16;
pub mod floats;
pub mod ipc;
pub mod json;
pub mod list;
pub mod memory;
pub mod scalar;
pub mod stream;
pub mod r#struct;
```

Each module addresses a gap in `arrow-rs`. The lib root itself has trait extensions.

### 2.2 The `DataTypeExt` and `RecordBatchExt` traits (15 min)

Read lines 1–500 of [`rust/lance-arrow/src/lib.rs`](../rust/lance-arrow/src/lib.rs). The patterns to internalize:

- **`DataTypeExt`** — extension trait on `arrow_schema::DataType`:
  - `is_binary_like()`, `is_struct()`, `is_fixed_stride()`, `byte_width()`, `bit_width()`, `is_nested()` etc.
  - Read the whole trait. These predicates show up *everywhere* in encoding and scanning logic.
- **`RecordBatchExt`** — extension trait on `RecordBatch`:
  - `try_with_column`, `merge`, `drop_column`, `replace_column_by_name`. RecordBatch composition utilities.
- **`SchemaExt`** — extension trait on `arrow_schema::Schema`:
  - Field-name lookups, projection, merging.

The pattern: define a trait with all the methods you wish a foreign type had, then `impl YourTrait for TheirType {}`. Everywhere the foreign type is in scope, the new methods feel native.

### 2.3 bfloat16 — `lance-arrow/src/bfloat16.rs` (5 min)

`bf16` is critical for ML embeddings: half the bytes of `f32`, same exponent range, 1/2 the mantissa bits. arrow-rs doesn't have a first-class `bf16` array type, so Lance defines it as an Arrow extension type backed by `FixedSizeBinary`.

Skim the file. The concept is: write a logical bfloat16 column, but store it as 2-byte FSB on disk.

### 2.4 deepcopy — `lance-arrow/src/deepcopy.rs` (5 min)

Arrow arrays are normally cheap to clone (they're `Arc<ArrayData>` underneath). But sometimes you need to physically copy the buffers — e.g., to free memory mapped from a file before closing the file. That's what `deepcopy` does.

You'll see it called in places where Lance can't hold onto memory-mapped buffers across an `await` point.

### 2.5 IPC — `lance-arrow/src/ipc.rs` (10 min)

Arrow IPC is a binary serialization format for Arrow record batches. Lance uses it to serialize:

- Per-fragment statistics (max/min/null-count) for pushdown
- Some index-related auxiliary data
- Test fixtures

Read the header of `ipc.rs` to see the wrapper API. The actual encoding is `arrow-ipc`.

### 2.6 stream — `lance-arrow/src/stream.rs` (10 min)

Adapters between `Stream<Item = RecordBatch>`, `Iterator<Item = RecordBatch>`, and `RecordBatchReader`. Lance's APIs accept any of these via thin shims, all defined here.

### 2.7 JSON — `lance-arrow/src/json.rs` (10 min)

Conversion between Arrow values and JSON, used by the SQL UDTF integration and some debug paths. You probably won't touch this often; just know where it is.

**End of Block 2.** Stretch.

## Block 3 — `lance-io` (75 min)

This is the I/O bedrock: object stores, readers/writers, and the scheduler that batches and prioritizes I/O across many concurrent scans.

### 3.1 Crate overview — `lance-io/src/lib.rs` (10 min)

Open [`rust/lance-io/src/lib.rs`](../rust/lance-io/src/lib.rs) and read it fully (~280 lines).

The headline type is **`ReadBatchParams`**:

```rust
pub enum ReadBatchParams {
    Range(Range<usize>),
    Ranges(Arc<[Range<u64>]>),
    RangeFull,
    RangeTo(RangeTo<usize>),
    RangeFrom(RangeFrom<usize>),
    Indices(UInt32Array),
}
```

This describes "which rows I want from a file" — a contiguous range, multiple ranges, all rows, or scattered indices. Every read API in Lance takes a `ReadBatchParams`. Read the `slice`, `to_offsets`, and `to_ranges` methods to feel the conversions.

### 3.2 ObjectStore — `lance-io/src/object_store.rs` (20 min)

Open [`rust/lance-io/src/object_store.rs`](../rust/lance-io/src/object_store.rs). At 56 KB it's a lot, but you only need an outline:

1. Lines 1–100: imports and constants. Notice the parallelism constants (`DEFAULT_LOCAL_IO_PARALLELISM = 8`, `DEFAULT_CLOUD_IO_PARALLELISM = 64`) — local disks need few threads, cloud stores need many.
2. Find the `pub struct ObjectStore` declaration. Read its fields. It wraps an `Arc<DynObjectStore>` (the foreign trait object from the `object_store` crate) plus Lance-specific extras: tracing, throttling, credential refresh, IO stats.
3. Read `ObjectStore::from_uri` or `ObjectStore::from_uri_and_params`. This is the entry point — given a URI like `s3://bucket/dataset.lance` or `/local/path`, it picks the right backend.
4. Read `ObjectStore::open` and `ObjectStore::create_dir_all`, `ObjectStore::delete_dir_all`. These are the high-level operations.

Then look at the `providers/` submodule:

```bash
ls rust/lance-io/src/object_store/providers/
```

Each provider (local, memory, AWS, Azure, GCS, OSS, HuggingFace) is a self-contained adapter that implements `ObjectStoreProvider`. The `ObjectStoreRegistry` lets users plug in custom providers. This is how Lance handles every cloud — you don't write conditionals, you register a provider.

### 3.3 Reader and Writer traits — `lance-io/src/traits.rs` (5 min)

Read the file fully (~150 lines). Two traits:

- **`Reader`** — `async fn get_range`, `async fn get_all`, `block_size()`, `path()`. Every "thing we can read bytes from" implements this.
- **`Writer`** — `async fn write_all`, `async fn shutdown`. Symmetric.

Every concrete reader/writer wraps an `Arc<dyn ObjectStore>` plus a `Path`. Reads call `object_store.get_opts(path, range)` under the hood.

### 3.4 ObjectReader — `lance-io/src/object_reader.rs` (10 min)

Open [`rust/lance-io/src/object_reader.rs`](../rust/lance-io/src/object_reader.rs). Read lines 1–200.

**`CloudObjectReader`** is the cloud reader: long-lived, supports parallel range gets, retries, and prefetching for the head/tail of a file (the Lance footer is at the end, the manifest pointer is in the footer — so we always do at least one tail-fetch per file).

**`SmallReader`** is for files small enough to read into memory once.

`LocalObjectReader` (in `local.rs`) is local-disk-specific and on Linux can use io-uring under the hood.

### 3.5 The Scheduler — `lance-io/src/scheduler.rs` (20 min)

Open [`rust/lance-io/src/scheduler.rs`](../rust/lance-io/src/scheduler.rs). At 60 KB this is a beast; read with discipline.

**Why a scheduler exists.** A naïve scan would issue one I/O per range and let `tokio` schedule them. That would: (a) overwhelm cloud rate limits, (b) miss opportunities to coalesce nearby ranges into one request, (c) starve high-priority requests behind low-priority ones, (d) saturate memory by reading too far ahead.

The scheduler addresses all four. It accepts I/O requests with priorities, coalesces nearby ranges into bigger reads, applies backpressure based on in-flight bytes, and dispatches in priority order via a binary heap.

What to read:

1. Lines 1–80: imports, `BACKPRESSURE_*` constants, global IOPS/bytes counters.
2. Find the `pub struct ScanScheduler`. Note its `BinaryHeap` of pending requests, the `AtomicU64` stats, and the `Notify` used for backpressure.
3. Find `ScanScheduler::new` and `ScanScheduler::open_file` — entry points.
4. Find `submit_request` (on `FileScheduler`, the per-file handle) — the "give me these ranges with this priority" API.

Then open [`rust/lance-io/src/scheduler/lite.rs`](../rust/lance-io/src/scheduler/lite.rs). This is a lighter-weight variant used for cases where the full scheduler's machinery is overkill (small reads, in-memory tests, single-shot operations). Skim it; the API is parallel.

**Key insight:** the scheduler implements `EncodingsIo` (the trait you saw on day 1 in `lance-encoding/src/lib.rs`). That trait is the contract between `lance-io` and `lance-encoding`: a "thing that can fetch byte ranges asynchronously, with priorities." `lance-encoding` doesn't depend on `lance-io` directly; it depends on the trait.

### 3.6 ObjectWriter — `lance-io/src/object_writer.rs` (5 min)

Read the top of the file. Two writers:

- `ObjectWriter` for cloud (multipart upload, with retries on failure).
- `LocalWriter` for local disk (single-syscall writes, buffered).

Per `rust/AGENTS.md`: local writes should *not* reuse the multipart machinery. The two paths exist for performance reasons.

### 3.7 Encodings — `lance-io/src/encodings/` (5 min)

There are three legacy file-level encodings here: `plain.rs`, `binary.rs`, `dictionary.rs`. These are remnants of the v1 file format and are still supported for reading older files. The "real" encodings live in `lance-encoding` (day 3).

You can ignore this directory unless you're investigating compatibility with old files.

### 3.8 io-uring — `lance-io/src/uring/` (skip; Linux-only optimization)

io-uring is a Linux-specific async syscall API. Lance has an experimental backend that bypasses tokio's blocking-syscall pool. **Off by default, gated behind the `io-uring` feature.** You can ignore it for the curriculum.

**End of Block 3.** Stretch.

## Block 4 — Hands-on: trace `Dataset::open` (30 min)

Pick a real entry point and follow it through the three crates you just read. This is the single best exercise to cement the day.

### Setup

Open these files in tabs (or splits):

1. [`rust/lance/src/lib.rs`](../rust/lance/src/lib.rs)
2. [`rust/lance/src/dataset.rs`](../rust/lance/src/dataset.rs)
3. [`rust/lance/src/dataset/builder.rs`](../rust/lance/src/dataset/builder.rs)
4. [`rust/lance-table/src/io/manifest.rs`](../rust/lance-table/src/io/manifest.rs)
5. [`rust/lance-io/src/object_store.rs`](../rust/lance-io/src/object_store.rs)

### Trace

Start at `lance::open_dataset` in `lib.rs`. Follow the path:

```
open_dataset(uri)
  └── DatasetBuilder::from_uri(uri).load().await
        ├── ObjectStore::from_uri(uri)              // lance-io
        ├── CommitHandler::resolve_latest_location  // lance-table
        ├── read_manifest(object_store, path)        // lance-table::io::manifest
        │     ├── object_store.open(path)            // lance-io::object_reader
        │     ├── reader.get_range(footer_offset..end)  // tail read
        │     ├── parse the magic bytes              // "LANC" check
        │     └── decode the protobuf Manifest        // prost
        ├── construct Schema from Manifest
        ├── build the per-dataset caches
        └── return Dataset { object_store, manifest, ..., }
```

You won't read every line. Goal:

1. Find each step in the source. Use `rg` if needed (`rg "fn from_uri" rust/lance/src/dataset/`).
2. Note **which crate each step lives in** and **which type it returns**.
3. When you hit something unfamiliar (e.g., `CommitHandler`), don't dive — write down the question and continue. Day 4 covers `lance-table` in depth.

### Verify with a debugger or print statements

If you're comfortable, add a single `tracing::info!("opened manifest at {}", ...)` somewhere on the path, recompile, and run `cargo run -p lance-examples --example write_read_ds`. Watching one log line on a real run is worth ten readings.

(If you do this, **revert before moving on**. Don't merge debug prints — see `AGENTS.md`.)
