### 1.2 Manifest — `lance-table/src/format/manifest.rs` (15 min)

Open the file. It's ~50 KB but you only need a tour.

```rust
pub struct Manifest {
    pub schema: Schema,             // dataset schema (lance-core Schema)
    pub version: u64,               // monotonically increasing version number
    pub branch: Option<String>,     // None for main
    pub writer_version: Option<WriterVersion>,
    pub fragments: Arc<Vec<Fragment>>,
    pub version_aux_data: usize,    // offset in this manifest file
    pub index_section: Option<usize>,
    // ... timestamps, format version, feature flags, base paths, ...
}
```

The manifest is the **complete description of the dataset at a point in time**. Every commit produces a new manifest. The directory `_versions/N.manifest` is just a serialized `Manifest` proto.

Things to find in the source:

1. The conversion `impl From<&Manifest> for pb::Manifest` (write side) and `impl TryFrom<pb::Manifest> for Manifest` (read side) — search for `impl TryFrom<pb::Manifest>`.
2. Methods like `Manifest::add_fragments`, `Manifest::with_indices`, `Manifest::feature_flags`. These are **immutable transforms** — Lance never mutates a manifest in place; it builds a new one.

The manifest file itself is a small Lance file (it uses `MAGIC = "LANC"` too). See `lance-table/src/io/manifest.rs` for `read_manifest` and `write_manifest`.

### 1.3 Fragment & DataFile — `lance-table/src/format/fragment.rs` (15 min)

Open [`rust/lance-table/src/format/fragment.rs`](../rust/lance-table/src/format/fragment.rs). Read lines 1–250.

Two related types:

**`DataFile`**:
```rust
pub struct DataFile {
    pub path: String,             // relative to dataset
    pub fields: Arc<[i32]>,       // field IDs stored in this file
    pub column_indices: Arc<[i32]>, // column position per field id (-1 = none)
    pub file_major_version: u32,
    pub file_minor_version: u32,
    pub file_size_bytes: CachedFileSize,
    pub base_id: Option<u32>,     // for external base paths
}
```

A `DataFile` is a Lance v2 file (the file format you learned yesterday). One fragment **may have multiple data files**, each holding a subset of fields. This is the core idea behind cheap column addition: a new column gets a new data file in each fragment instead of rewriting the existing data.

**`Fragment`**:
```rust
pub struct Fragment {
    pub id: u64,
    pub files: Vec<DataFile>,
    pub deletion_file: Option<DeletionFile>,
    pub physical_rows: Option<usize>,
    pub row_id_meta: Option<RowIdMeta>,
    // ...
}
```

A fragment is one row-range chunk of the dataset. Each fragment has:

- A stable `id`
- One or more `DataFile`s (typically one per "data version" you've added columns at)
- Optional deletion info — a `RoaringBitmap` of deleted row offsets, persisted in `_deletions/` and referenced by the manifest.
- `physical_rows` — how many rows are physically stored (including deleted ones).
- `row_id_meta` — info needed for the stable row id index (see §1.6 below).

Per `rust/AGENTS.md`: never use raw `physical_rows` for user-visible "row count" metrics — subtract deletions, use `Fragment::num_rows()`.

### 1.4 IndexMetadata — `lance-table/src/format/index.rs` (5 min)

Open [`rust/lance-table/src/format/index.rs`](../rust/lance-table/src/format/index.rs) and read it fully (~12 KB).

```rust
pub struct IndexMetadata {
    pub uuid: Uuid,
    pub name: String,
    pub fields: Vec<i32>,
    pub dataset_version: u64,
    pub fragment_bitmap: Option<RoaringBitmap>,
    pub index_details: Option<prost_types::Any>,
    pub index_version: i32,
    pub created_at: Option<DateTime<Utc>>,
    pub base_id: Option<u32>,
}
```

Indices are stored separately under `_indices/<uuid>/` and referenced from the manifest via `IndexMetadata`. The `fragment_bitmap` says "this index covers these fragments." When a new fragment is appended, the index doesn't immediately cover it — until a maintenance step adds it. (This is one source of subtle bugs in production; remember it.)

`index_details` is a `prost_types::Any` — it's a serialized proto specific to the index type, containing parameters like IVF partition count, PQ subvector count, etc.

### 1.5 Transaction — `lance-table/src/format/transaction.rs` + `protos/transaction.proto` (15 min)

Open [`protos/transaction.proto`](../protos/transaction.proto) and read it in full. Skim it before reading; the comments are good.

```protobuf
message Transaction {
  uint64 read_version = 1;
  string uuid = 2;
  // ...
  message Append { ... }
  message Delete { ... }
  message Overwrite { ... }
  message Update { ... }
  message Merge { ... }
  message ReserveFragments { ... }
  message Project { ... }
  message Restore { ... }
  message UpdateConfig { ... }
  // ... more operation types
}
```

A `Transaction` is the **description of a single change** to the dataset. It carries:

- `read_version`: what version was read while computing this change. Used for conflict detection.
- `uuid`: idempotency key. Re-submitting the same UUID should be safe.
- An operation variant (Append, Delete, Overwrite, etc.) carrying the per-operation payload.

Why does Lance store the transaction proto separately (in `_transactions/`) when the new manifest already contains the result? Two reasons:

1. **Conflict resolution.** When two writers race to commit, the loser needs to know what the winner did. By reading the winner's transaction, the loser can decide: "is my change still valid? do I need to retry?"
2. **Audit and replay.** Transactions form a log of intent.

Now open the (very small) Rust file [`rust/lance-table/src/format/transaction.rs`](../rust/lance-table/src/format/transaction.rs) — it's mostly a thin wrapper with conversions.

The Lance-side `Transaction` struct (the rich, behavior-bearing one) actually lives at [`rust/lance/src/dataset/transaction.rs`](../rust/lance/src/dataset/transaction.rs) (~244 KB). You'll meet it in block 4.

### 1.6 Row IDs — `lance-table/src/rowids/` (10 min)

Open [`rust/lance-table/src/rowids.rs`](../rust/lance-table/src/rowids.rs) and skim. Then look at the submodule layout:

```
rowids/
├── bitmap.rs
├── encoded_array.rs
├── index.rs
├── segment.rs
├── serde.rs
└── version.rs
```

The big ideas:

- **Stable Row IDs are opt-in** (controlled by the `FLAG_STABLE_ROW_IDS` feature flag in the manifest). When enabled, every row gets a stable 64-bit `RowId` that survives compaction.
- The mapping from `RowId` → `RowAddress` (fragment + offset) is stored as a *set of segments* per fragment.
- Segments are run-length-encoded (`encoded_array.rs`) for compactness; bitmaps (`bitmap.rs`) are used for masks.
- The whole index is loaded lazily and cached.

You don't need to read all of this today. Know it exists; come back when you debug something rowid-related.

### 1.7 Commit handlers — `lance-table/src/io/commit.rs` (10 min)

Open [`rust/lance-table/src/io/commit.rs`](../rust/lance-table/src/io/commit.rs). Read the module-level doc comment (top ~30 lines):

> In Lance, a transaction is committed by writing the next manifest file. However, care should be taken to ensure that the manifest file is written only once, even if there are concurrent writers...

Then look for these:

- **`trait CommitHandler`** — the polymorphic interface. Implementations decide *how* to make "write this manifest at version N+1" atomic.
- **`RenameCommitHandler`** — default for most stores (write to `tmp`, rename to final; atomic on local filesystems and most cloud stores **except S3**).
- **`UnsafeCommitHandler`** — used on S3 by default. It's "unsafe" because S3 historically lacked atomic put-if-not-exists. (Modern S3 does, and Lance is moving toward using it, but the legacy default is still here.)
- **`ExternalManifestCommitHandler`** — uses an external coordination service (DynamoDB, etc.) to serialize commits.
- **`CommitLock`** — simpler trait when you just want to provide a mutex.
- **`ManifestNamingScheme`** — V1 uses `_versions/N.manifest`; V2 uses `_manifests/(MAX-N).manifest` so listing returns the latest version first (efficient on object stores).

Different storage backends + concurrency requirements pick different strategies; that's why this is pluggable.

### 2.3 Sessions and caches — `lance/src/session.rs` (10 min)

Open [`rust/lance/src/session.rs`](../rust/lance/src/session.rs). Read lines 1–100.

A `Session` holds **shared state across multiple datasets** in the same process: the metadata cache, the index cache, the index registry, and the namespace storage options provider. You can pass a `Session` to `DatasetBuilder::with_session(...)` to share caches between datasets.

Most users don't think about sessions. But when you load *many* datasets in one process, sharing a `Session` saves significant RAM and avoids redundant index loads.

## Block 3 — `Scanner` and the read pipeline (60 min)

The scanner is the thing that turns "read these rows with this filter and these projections" into a parallelized stream of `RecordBatch`es.

### 3.1 The Scanner builder (15 min)

Open [`rust/lance/src/dataset/scanner.rs`](../rust/lance/src/dataset/scanner.rs). At 423 KB this is the **biggest file in the repo.** Use the outline.

What to find:

1. The `pub struct Scanner` declaration (it's high in the file). Note the fields: `dataset`, `projection_plan`, `filter`, `nearest`, `limit`, `offset`, `prefilter`, `with_row_id`, `with_row_address`, `batch_size`, `fragments`, etc.
2. The `pub fn` builder methods: `project`, `project_with_transform`, `filter`, `nearest`, `limit`, `with_row_id`, `with_fragments`, `prefilter`, `batch_size`, `use_scalar_index`, `use_index`, `fast_search`, `nprobes`, `refine_factor`, etc. **Each one is small** — they all just set a field and return `&mut self`. Read 5–6 to feel the pattern.
3. `Scanner::try_into_stream` — the "go" button. Returns `Result<DatasetRecordBatchStream>`.
4. `Scanner::try_into_batch` (test/test-adjacent helper) — collects the stream into a single batch.
5. `Scanner::create_plan` (or similarly named) — constructs the DataFusion physical plan that backs the scan.

The scanner is **declarative until you call `try_into_stream`.** That call walks the builder state, picks the best execution path (use a vector index? a scalar index? full scan? prefilter?), and constructs a DataFusion `ExecutionPlan`.

### 3.3 Scan vs Take (10 min)

Two complementary primitives:

- **Scan** — read columns of many rows in row order. Cheap per row when many rows are read.
- **Take** — gather *specific* row IDs (or row offsets). Used after kNN ("the top-k rows are these IDs; now read their values"). Higher per-row cost; cheap because you only do it for k rows.

Lance's vector search pipeline always ends in a Take: the index returns row IDs/distances; the scanner takes those rows from disk to project the requested columns.

This is also why random access matters in Lance: the Take is the hot path on every vector search.

### 3.4 `Scanner::nearest` — vector search end-to-end (10 min)

Find `Scanner::nearest` in `scanner.rs`. It accepts a `Query`:

```rust
pub struct Query {
    pub column: String,
    pub key: Arc<dyn Array>,  // the query vector
    pub k: usize,
    pub nprobes: usize,
    pub refine_factor: Option<u32>,
    pub metric_type: MetricType,
    pub use_index: bool,
    // ...
}
```

When `use_index = true` and an index covers the column, the plan becomes:

```
TakeExec
 └ KNNVectorDistanceExec (refine: re-rank top-k * refine_factor with exact distances)
   └ ScalarIndexExec (if you also have a filter and prefilter=true)
   └ ANN scan (IVF traversal — see lance-index)
```

When no index, it's a brute-force flat scan + KNN.

### 4.2 Append/Overwrite via `Insert` (10 min)

Open [`rust/lance/src/dataset/write/insert.rs`](../rust/lance/src/dataset/write/insert.rs). Read lines 1–150.

Path:

1. The user passes a `RecordBatchReader` (or anything that streams `RecordBatch`es).
2. Lance breaks the stream into row groups (`break_stream` from `lance-datafusion::chunker`) and target-sized chunks (`chunk_stream`).
3. Each chunk is written as a `Fragment` with one `DataFile` (a Lance v2 file). This is `write_fragments_internal` — find it; it's the workhorse.
4. The fragments accumulate.
5. A `Transaction` of type `Append` or `Overwrite` is built referencing the new fragments.
6. The transaction is committed via the `CommitHandler` (writes the new manifest).

For an append, the new fragments are **added** to the existing fragment list. For an overwrite, the new fragment list replaces the old one (old data files are not deleted right away — they're cleaned up later via `cleanup_old_versions`, supporting time travel).

### 4.3 MergeInsert (UPSERT) (10 min)

Open [`rust/lance/src/dataset/write/merge_insert.rs`](../rust/lance/src/dataset/write/merge_insert.rs). It's 330 KB; just orient yourself.

`MergeInsert` is the SQL `MERGE INTO` / "upsert" operator. Roughly:

```rust
let mut builder = MergeInsertBuilder::new(dataset, ["id"]); // join on id
builder
    .when_matched(WhenMatched::UpdateAll)
    .when_not_matched(WhenNotMatched::InsertAll)
    .execute(stream).await?;
```

Internally this:

1. Joins the input stream against existing rows using a `HashJoinExec`.
2. For matched rows: produces an updated row in a new fragment, marks the old row as deleted.
3. For unmatched rows: writes them as new fragments.
4. Builds a `Transaction::Update` (or `MergeInsert`-specific variant) that records both the new fragments and the deletion vectors to add.
5. Commits.

The submodule `merge_insert/exec/` has DataFusion `ExecutionPlan`s specific to merge-insert (e.g., for the "assign action: update vs insert" stage).

### 4.4 Delete and Update (5 min)

`write/delete.rs`:

- A `DeleteBuilder` accepts a SQL predicate (`category = 'a' AND id < 100`).
- Lance evaluates the predicate against each fragment, building a per-fragment `RoaringBitmap` of deleted offsets.
- Bitmaps are written to `_deletions/<fragment_id>-<seq>.deletion` files (or to the manifest for tiny ones).
- A `Transaction::Delete` is committed.

`write/update.rs`:

- An `UpdateBuilder` accepts a predicate and a set of column-expression pairs.
- Lance evaluates the predicate, then for each matched row builds the new column values via DataFusion, writes affected fragments with new data files for the updated columns, and (for changed identifying columns) marks rows as deleted/inserted.

These are surprisingly subtle; the implementations are large because they handle indices, deletion vectors, blob columns, and schema evolution all together.

### 4.5 Commit (10 min)

Open [`rust/lance/src/dataset/write/commit.rs`](../rust/lance/src/dataset/write/commit.rs).

```rust
pub struct CommitBuilder { ... }
```

The builder constructs and commits a `Transaction` to a target dataset/URI. Internal flow:

1. Choose the next version number = current_version + 1.
2. Apply the transaction's operation to the current manifest (in memory) to produce the new manifest.
3. Use the `CommitHandler` to write the new manifest atomically.
4. On conflict (the chosen version was taken by a concurrent commit), retry: re-read the manifest, recompute the operation if needed (e.g., re-evaluate a delete predicate against the new state), pick the next version, and try again.

This is the **distributed coordination heart** of Lance. The retry loop and conflict-resolution policy are the most subtle code in the system. Don't try to master it today; know it exists.

The actual commit machinery lives in `rust/lance/src/io/commit.rs` (78 KB). Skim its top-level functions: `commit_new_dataset`, `commit_transaction`, `commit_detached_transaction`, `detect_overlapping_fragments`.
