# Day 1, Afternoon — Project Tour

Time: ~2 hours. Goal: build a mental map of the whole codebase so that, on
later days, you know where to look without searching.

The trick to learning a large project is **always read top-down**. We'll start
at the workspace, descend to the Rust crate, walk the module tree, and
finish with a hands-on run of the simple example.

---

## 1. The workspace

Open `Cargo.toml` (the one at the repo root):

```toml
[workspace]
members = ["rust/lancedb", "nodejs", "python"]
resolver = "2"

[workspace.dependencies]
lance = { version = "=7.0.0-beta.7", git = "https://github.com/lance-format/lance.git", ... }
lance-core = ...
lance-index = ...
arrow = "58.0.0"
arrow-array = "58.0.0"
datafusion = { version = "53.0.0", default-features = false }
...
```

Things to take from this:

* The repo is a **Cargo workspace** with three Rust members. (`java` builds
  separately with Maven and isn't part of the workspace.)
* It depends heavily on **Lance** (a sibling project; pinned to a single
  version). When you wonder "where does this column actually get written?",
  the answer is usually "the `lance` crate".
* Versions of Arrow and DataFusion are **workspace-wide** so all three Rust
  crates use the same versions. This matters because Arrow has frequent
  releases and `RecordBatch` from one version is incompatible with another.

Key Lance sub-crates you'll see imported:

| Crate | What it does (1-line) |
|-------|----------------------|
| `lance`           | Top-level dataset API (`Dataset`, `WriteParams`, `ReadParams`). |
| `lance-core`      | Core types shared across the family.                            |
| `lance-file`      | Lance file format read/write.                                   |
| `lance-table`     | Table-level concerns: manifests, commit handlers.               |
| `lance-index`     | Index implementations (IVF, HNSW, BTree, FTS, scalar). |
| `lance-io`        | Object store wrappers, IO scheduling.                           |
| `lance-linalg`    | Distance metrics (`L2`, `Cosine`, `Dot`, `Hamming`).            |
| `lance-encoding`  | Column encodings, file version selection.                       |
| `lance-datafusion`| Bridges Lance ↔ DataFusion (so you can scan a Lance dataset as a DataFusion table). |
| `lance-arrow`     | Arrow extension utilities (e.g. `RecordBatchExt`).              |
| `lance-namespace` | Namespace abstraction (a tree of catalogs).                     |
| `lance-namespace-impls` | Concrete namespace impls (REST, dir, ...).                |

Don't try to memorize. When you see `use lance_index::scalar::FullTextSearchQuery`
later, you'll be able to predict that's "the FTS query type, lives in the
indexing crate".

## 3. The module tree (your map)

This is the most important section of the chapter. Read it once, and refer
back as needed.

```
rust/lancedb/src/
├── lib.rs                  # Top-level: re-exports, `DistanceType`, top-level docs.
│
├── connection.rs           # `connect()`, `ConnectBuilder`, `Connection`,
├── connection/             #   the user-facing top-level handle. Holds an
│   └── create_table.rs     #   Arc<dyn Database>. Sub-builders for create_table.
│
├── database.rs             # `Database` trait + requests (`OpenTableRequest` etc).
├── database/
│   ├── listing.rs          # `ListingDatabase` — local file/object-store backed.
│   └── namespace.rs        # `LanceNamespaceDatabase` — talks to a Lance namespace.
│
├── table.rs                # The big one (~146 KB).
│   │                       # `Table` (public struct), `BaseTable` (trait),
│   │                       # `NativeTable` (local impl), `TableDefinition`,
│   │                       # `WriteOptions`, `Filter`, `Tags`, `ColumnKind`...
├── table/
│   ├── dataset.rs          # `DatasetConsistencyWrapper` — read consistency.
│   ├── add_data.rs         # `AddDataBuilder`, append/overwrite logic.
│   ├── update.rs           # `UpdateBuilder`.
│   ├── delete.rs           # `delete()` plumbing.
│   ├── merge.rs            # `MergeInsertBuilder` (UPSERT).
│   ├── optimize.rs         # `OptimizeAction`, compaction.
│   ├── schema_evolution.rs # add_columns / alter_columns / drop_columns.
│   ├── query.rs            # `NativeTable` query implementation.
│   ├── write_progress.rs   # progress reporting on writes.
│   └── datafusion/         # custom DataFusion ExecutionPlans (e.g. InsertExec).
│
├── query.rs                # The big query module: `Query`, `VectorQuery`,
│                           # `Select`, `IntoQueryVector`, `ExecutableQuery`,
│                           # `QueryBase`, hybrid execution.
├── query/
│   └── hybrid.rs           # Hybrid (vector + FTS) merging logic.
│
├── index.rs                # `Index` enum + `IndexBuilder` (`create_index`).
├── index/
│   ├── vector.rs           # IVF_PQ, HNSW, IVF_FLAT, IVF_RQ, IVF_HNSW_*  builders.
│   ├── scalar.rs           # BTree, Bitmap, LabelList, FTS builders.
│   └── waiter.rs           # `wait_for_index` helper.
│
├── embeddings.rs           # `EmbeddingFunction` trait + registry; `EmbeddingDefinition`.
├── embeddings/
│   ├── openai.rs           # OpenAI text-embedding (feature `openai`).
│   ├── bedrock.rs          # AWS Bedrock (feature `bedrock`).
│   └── sentence_transformers.rs  # Local HF transformers (feature `sentence-transformers`).
│
├── data.rs                 # Re-export module.
├── data/
│   ├── scannable.rs        # `Scannable` trait — what we accept as input data.
│   ├── sanitize.rs         # Coerce input batches to match table schema.
│   └── inspect.rs          # Peek at first batch to infer schema.
│
├── arrow.rs                # `RecordBatchStream`/`Reader`, glue to DataFusion streams.
├── ipc.rs                  # Arrow IPC encode/decode (used by remote).
│
├── expr.rs                 # `col(...)`, `lit(...)` helpers (DataFusion expressions).
├── expr/
│   └── sql.rs              # SQL string ↔ Datafusion `Expr` parsing.
│
├── rerankers.rs            # `Reranker` trait, normalization helpers.
├── rerankers/
│   └── rrf.rs              # Reciprocal Rank Fusion.
│
├── dataloader.rs           # Re-export.
├── dataloader/
│   └── permutation/        # Permutation-based batch shuffler (for ML training).
│
├── io.rs                   # Re-export.
├── io/
│   └── object_store.rs     # Build Lance `ObjectStoreParams` from connection options.
│
├── remote.rs               # Re-exports for `[cfg(feature = "remote")]`.
├── remote/
│   ├── client.rs           # `RemoteClient`, `ClientConfig`, headers, auth.
│   ├── retry.rs            # Retry policy (request/connect/read failure budgets).
│   ├── db.rs               # `RemoteDatabase` (impl of `Database`).
│   ├── table.rs            # `RemoteTable` (impl of `BaseTable`) ~217 KB.
│   └── util.rs             # Small helpers (URL building etc).
│
├── utils/
│   ├── mod.rs              # Misc helpers; `MaxBatchLengthStream`, `TimeoutStream`.
│   └── background_cache.rs # Time-based cache that refreshes in the background (uses moka indirectly via custom logic + Shared futures).
│
├── polars_arrow_convertors.rs  # Polars ↔ Arrow conversion (feature `polars`).
└── test_utils.rs / test_utils/  # Helpers used by integration tests.
```

A few mental sub-groupings:

* **"Plumbing for I/O & data"**: `arrow.rs`, `ipc.rs`, `data/`, `io/`,
  `dataloader/`, `polars_arrow_convertors.rs`, `utils/`.
* **"Catalog layer"**: `connection.rs`, `database.rs`, `database/listing.rs`,
  `database/namespace.rs`, `remote/db.rs`.
* **"Table layer"**: `table.rs`, `table/*`, `remote/table.rs`.
* **"Query/Index layer"**: `query.rs`, `query/hybrid.rs`, `index.rs`,
  `index/*`, `expr/*`, `rerankers/*`.
* **"Optional features"**: `embeddings/openai.rs`, `embeddings/bedrock.rs`,
  `embeddings/sentence_transformers.rs`, `polars_arrow_convertors.rs`.

Print this map or keep it open in a tab on Days 2–4.

## 4. The Python crate (skim only today)

`python/` is a separate Cargo crate (`python/Cargo.toml`) that depends on
`lancedb` and uses **PyO3** to expose it as a Python extension module.

```
python/
├── src/                       # Rust side of the bindings (PyO3 wrappers)
│   ├── lib.rs                 # `#[pymodule] fn _lancedb(...)`. Registers Python classes.
│   ├── connection.rs          # `Connection` Python class.
│   ├── table.rs               # `Table` Python class.
│   ├── query.rs               # Query / VectorQuery / FTSQuery / HybridQuery.
│   ├── index.rs               # Index config.
│   ├── embeddings/...         # (none — embeddings are Python-only on this side)
│   ├── runtime.rs             # Background tokio runtime for async-from-sync.
│   ├── error.rs               # Map lancedb::Error → Python exceptions.
│   └── ...
└── python/lancedb/             # Pure-Python wrapper layer the user actually imports
    ├── __init__.py             # `connect(...)`, top-level entry point.
    ├── db.py                   # Sync + Async DB classes (LanceTable, AsyncTable).
    ├── table.py                # 190 KB. Sync + Async Table classes.
    ├── query.py                # Python-side Query / VectorQuery / FTSQuery.
    ├── index.py                # Index parameter classes.
    ├── embeddings/             # Many Python-only embedding functions (HuggingFace,
    │                           # OpenAI, Cohere, Gemini, Bedrock, Voyage, ...).
    ├── rerankers/              # Many Python-only rerankers.
    ├── pydantic.py             # Pydantic ↔ Arrow schema bridge.
    ├── _lancedb.pyi            # Type stubs for the Rust-implemented module.
    └── remote/                 # Python-side wrapper around RemoteTable / RemoteDatabase.
```

Pattern to internalize: the **Rust core is thin**, the **Python wrapper is
thick**. Embeddings and rerankers are mostly implemented in Python because
the ML ecosystem is in Python. The Rust core only has three first-class
embedding implementations (OpenAI, Bedrock, sentence-transformers). The Python
side has many more.

We come back to bindings on Day 4.

## 5. The Node.js crate (skim only today)

`nodejs/` is similar in structure:

```
nodejs/
├── src/                       # Rust side (napi-rs)
│   ├── lib.rs                 # `ConnectionOptions`, `OpenTableOptions`, module init.
│   ├── connection.rs
│   ├── table.rs
│   ├── query.rs
│   ├── index.rs
│   └── ...
└── lancedb/                    # TypeScript wrapper layer
    ├── index.ts
    ├── connection.ts           # `Connection`, `connect()`.
    ├── table.ts                # Abstract `Table`.
    ├── native_table.ts         # `LocalTable` (also handles remote via napi).
    ├── query.ts                # Query / VectorQuery / FTSQuery / HybridQuery.
    ├── arrow.ts                # Arrow IPC ↔ JS bridge (54 KB).
    ├── embedding/              # JS embedding implementations.
    └── rerankers/              # JS rerankers.
```

Same pattern: thin Rust binding layer, thicker TypeScript layer.

`napi-rs` is to Node what PyO3 is to Python: a way to expose a Rust crate as
a native Node addon. We'll look at it on Day 4.

## 6. Examples and tests

Tests come in three flavors:

* **Unit tests inline** in the same file (`#[cfg(test)] mod tests`). Most
  modules have these.
* **Integration tests** in `rust/lancedb/tests/` — separate binaries that link
  the crate as a library and exercise the public API end-to-end.
* **Doc tests**: code blocks in `///` comments. They're compiled and run by
  `cargo test`.

Examples in `rust/lancedb/examples/`:

* `simple.rs` — basic CRUD + vector search.
* `ivf_pq.rs` — vector index with custom params.
* `full_text_search.rs` — BM25 FTS.
* `hybrid_search.rs` — vector + FTS combined (requires `sentence-transformers`).
* `openai.rs`, `bedrock.rs`, `sentence_transformers.rs` — embedding integrations.

## 7. Hands-on (~30 min)

Run the simple example and read its output:

```bash
cargo run --example simple
```

Expected:

* It creates a `data/sample-lancedb/` directory.
* Lists tables, creates one with 1000 rows × 128-dim vectors, adds 1000 more,
  builds an `IVF_PQ` index on the `vector` column, runs a `nearest_to([1.0;
  128])` query with `limit(2)`, prints the result `RecordBatch`, then deletes
  some rows and drops the table.

Then **change something small** to confirm your build pipeline:

1. In `examples/simple.rs`, change `limit(2)` to `limit(5)`. Re-run. Confirm
   you now get 5 rows in the printed batch.
2. Change `Index::Auto` to `Index::IvfPq(IvfPqIndexBuilder::default())`.
   Re-run. (You'll need `use lancedb::index::vector::IvfPqIndexBuilder;`.)
   It should still work — `Index::Auto` was already picking IVF_PQ.

Don't commit anything; revert with `git checkout
rust/lancedb/examples/simple.rs`.

## 8. Look at the data

After running `simple.rs`, open the output directory in your editor:

```
data/sample-lancedb/
└── my_table.lance/        (or whatever path the example used)
    ├── data/             # Lance fragment data files (the actual rows).
    ├── _versions/        # Manifests (one per version).
    ├── _indices/         # Built indices.
    └── _transactions/    # Transaction log entries.
```

A LanceDB *table* is just a directory of files in this layout. There's no
running server. This is identical to how SQLite works (one file ≈ one
database) — you can copy the directory, move it across machines, etc.

If you ever wonder where state lives, look at this directory. The "database"
in the local case is a parent directory containing one such subdirectory per
table. (`ls data/sample-lancedb/` after the example.)

## 9. Who calls whom — the call graph at 30,000 ft

A query like
```rust
table.query().nearest_to(&[1.0; 128])?.execute().await?
```
touches roughly:

```
Table::query()                                   (rust/lancedb/src/table.rs)
└── returns Query                                 (rust/lancedb/src/query.rs)
    └── .nearest_to()                             (Query → VectorQuery)
        └── .execute()                            (impl ExecutableQuery)
            └── BaseTable::create_plan(...)       (table.rs trait method)
                ├── NativeTable::create_plan      (table.rs / table/query.rs)
                │   └── lance::Dataset::scan(...) (lance crate)
                │       └── reads files via object_store
                └── RemoteTable::create_plan      (remote/table.rs)
                    └── reqwest POST .../query    (remote/client.rs)
                        └── parse Arrow IPC stream (arrow.rs / ipc.rs)
```

Two separate code paths, one trait, same user-visible types. This is the
single most important architectural fact of LanceDB. Internalize it now.
