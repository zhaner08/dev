## Block 1 — Vector indices (75 min)

### 1.1 The Index trait family (15 min)

Re-open [`rust/lance-index/src/lib.rs`](../rust/lance-index/src/lib.rs) — you read this on day 1 quickly. Now look at it with index knowledge:

```rust
#[async_trait]
pub trait Index: Send + Sync + DeepSizeOf {
    fn as_any(&self) -> &dyn Any;
    fn as_index(self: Arc<Self>) -> Arc<dyn Index>;
    fn as_vector_index(self: Arc<Self>) -> Result<Arc<dyn vector::VectorIndex>>;
    fn statistics(&self) -> Result<serde_json::Value>;
    async fn prewarm(&self) -> Result<()>;
    fn index_type(&self) -> IndexType;
    async fn calculate_included_frags(&self) -> Result<RoaringBitmap>;
}
```

`IndexType` (you saw the enum on day 1) tags every index. `is_scalar()`, `is_vector()`, `is_system()` partition the universe.

Then re-read the `IndexType::version()` method. **Index format versions matter** — if you bump the on-disk layout, older readers must reject the new format. The `target_partition_size()` for vector indices is the heuristic Lance uses to compute the IVF partition count.

Now open [`rust/lance-index/src/vector.rs`](../rust/lance-index/src/vector.rs). Read lines 1–100. The `VectorIndex` trait is the **vector-specific** super-set:

- `search(query, params, prefilter, metrics)` → returns top-k results.
- `prewarm()`, `is_loadable()`, `metric_type()`, etc.
- Returns `RecordBatch` with row IDs and `_distance` columns.

There's also `pub const DIST_COL: &str = "_distance";` — the magic column name that vector search results use.

### 1.2 The vector index module map (10 min)

`ls rust/lance-index/src/vector/`:

```
flat.rs    flat/        # exact (brute-force) search
ivf.rs     ivf/         # IVF (Inverted File) — partition-based
pq.rs      pq/          # PQ (Product Quantization) — compress vectors
sq.rs      sq/          # SQ (Scalar Quantization) — scalar bit-packing
bq.rs      bq/          # BQ (Binary Quantization) — 1-bit per dim
hnsw.rs    hnsw/        # HNSW (Hierarchical Navigable Small World) — graph
graph.rs   graph/       # generic graph machinery (used by HNSW)
kmeans.rs               # k-means used to train IVF centroids
quantizer.rs            # Quantizer trait — common to PQ/SQ/BQ
residual.rs             # residual computation (vector - centroid)
storage.rs              # vector storage abstractions
transform.rs            # vector transforms (rotation, normalization)
v3/                     # the V3 sub-index format used by IVF
shared/                 # cross-quantizer helpers
distributed/            # distributed index merging
utils.rs
```

The names you saw on day 1 in `IndexType` (`IvfFlat`, `IvfPq`, `IvfSq`, `IvfHnswFlat`, `IvfHnswPq`, `IvfHnswSq`, `IvfRq`) are **compositions** of these:

- **IVF** is always the outer partitioning. It splits N vectors into K partitions via k-means.
- The **inner** sub-index per partition can be `Flat` (full vectors), `PQ`, `SQ`, `BQ`, `RQ`, or `HNSW` (graph) — possibly with quantization layered on the HNSW edges.

So `IVF_HNSW_PQ` = "partition with IVF, build an HNSW graph in each partition, store vectors compressed with PQ."

### 1.3 IVF — the partitioning step (15 min)

Open [`rust/lance-index/src/vector/ivf.rs`](../rust/lance-index/src/vector/ivf.rs). Read lines 1–150.

Key types:

- **`IvfModel`** — the trained centroids (one Arrow array of `K` centroid vectors).
- **`Ivf`** — the runtime index: holds the model, knows how to assign a vector to its closest partition(s).

Training flow (under `ivf/builder.rs`, `ivf/shuffler.rs`, `vector/kmeans.rs`):

1. Sample N training vectors from the column.
2. Run k-means (`vector/kmeans.rs`) to get K centroids. K is chosen from `target_partition_size`.
3. Assign each row to its nearest centroid → produces the partition assignment.
4. **Shuffle** rows into per-partition streams (this is `ivf/shuffler.rs`, ~44 KB — a non-trivial external-memory shuffle that handles datasets larger than RAM).
5. Within each partition, build the chosen sub-index (Flat, PQ, HNSW, etc.).

Search flow:

1. Compute distance from the query to each centroid → pick the top `nprobes` partitions.
2. Search each chosen partition using its sub-index.
3. Merge results, optionally re-rank using exact distances (the `refine_factor` knob).

### 1.4 PQ — Product Quantization (15 min)

Open [`rust/lance-index/src/vector/pq.rs`](../rust/lance-index/src/vector/pq.rs). Read lines 1–150.

PQ compresses each vector by:

1. Splitting the D-dimensional vector into M *sub-vectors* (D/M dimensions each).
2. Independently k-means-clustering each sub-vector dimension into 256 codewords.
3. Storing each vector as M bytes (one byte per sub-vector, indexing into its codebook).

Trade-off: a 1024-dim float32 vector (4096 bytes) becomes 16 bytes (M=16) but search uses asymmetric distance computation against the codebook tables.

Files:

- `pq.rs` (top-level) — the trained PQ model.
- `pq/builder.rs` — training (k-means per sub-vector).
- `pq/storage.rs` — on-disk layout for PQ codes (~40 KB; lots of bit-fiddling for fast distance lookup).
- `pq/distance.rs` — the asymmetric distance computation kernel.
- `pq/transform.rs` — encode a query against the codebook into the lookup table.

PQ is the workhorse. Most production deployments use `IVF_PQ` or `IVF_HNSW_PQ`.

### 1.5 HNSW — graph-based ANN (10 min)

Open [`rust/lance-index/src/vector/hnsw.rs`](../rust/lance-index/src/vector/hnsw.rs) — only ~50 lines.

HNSW is a multi-layer skip-list-like graph. Search starts at the top layer (sparse, long edges) and greedily descends to the bottom layer (dense, short edges). Excellent recall/latency curve at the cost of build time and memory.

Files:

- `hnsw/builder.rs` — graph construction (~37 KB).
- `hnsw/index.rs` — runtime search.
- `hnsw/online.rs` — incremental updates (added per-batch).

In Lance, HNSW is always inside an IVF partition. This bounds graph size and lets multiple partitions be searched in parallel.

### 1.6 Other quantizers (10 min)

- **SQ (Scalar Quantization)** in `sq/` — quantizes each *dimension* into 4 or 8 bits. Less aggressive than PQ but preserves dimension semantics. Faster to compute distances.
- **BQ (Binary Quantization)** in `bq/` — 1 bit per dimension; Hamming distance for super-fast scoring. Used for the first-stage scan in two-stage search.
- **RQ (Residual Quantization)** — multi-stage residuals. Currently `IvfRq` is at index version 2 (vs PQ at 1). See `IndexType::version` for the bump.
- **Flat** in `flat/` — store the full vectors. Used inside IVF when storage is cheap relative to compute, or as a baseline.

You don't need to read these in detail today. Know that they're variations on "compress vectors so distance computation is fast and memory is small."

### 1.7 The end-to-end vector search path (5 min)

Putting it together for a query like `dataset.scanner(nearest={...}).to_table()`:

1. `Scanner::nearest(...)` records the query.
2. `Scanner::try_into_stream` builds a DataFusion plan.
3. The plan picks the right index (looking at `IndexType` and `fragment_bitmap`).
4. The plan node `KNNVectorDistanceExec` (from `rust/lance/src/io/exec/knn.rs`) loads the index, calls `VectorIndex::search`, gets a `RecordBatch` of `(_rowid, _distance)`.
5. A `TakeExec` then loads the projected columns by row id.
6. Results are sorted, limited, and returned.

The full pipeline lives in `rust/lance/src/index/vector.rs` (~126 KB) and `rust/lance/src/io/exec/knn.rs` (~95 KB). They're large; you'd visit them when investigating a specific query path.

## Block 2 — Scalar indices (60 min)

### 2.1 Scalar index trait — `lance-index/src/scalar.rs` (15 min)

Open [`rust/lance-index/src/scalar.rs`](../rust/lance-index/src/scalar.rs). Read lines 1–250.

Key abstractions:

- **`ScalarIndex`** — base trait. Methods: `search` (given a query, return matching `RowAddr`s), `update`, `remap`, `train`, `try_clone`.
- **`SargableQuery`** / **`AnyQuery`** / **`MaterializationStyle`** — describe what a query asks of an index. SARGable = "Search ARGument-able," predicates an index can answer.
- **`ScalarIndexCriteria`** — the "this index can answer this query if..." matching logic.

Most scalar indices output a `RowAddrTreeMap` (per-fragment bitmap) of row addresses that satisfy the predicate. The scanner uses this as the prefilter for a vector search or as a row mask for a regular scan.

### 2.2 Index families (30 min)

Each scalar index targets a different filter pattern.

#### BTree — `lance-index/src/scalar/btree.rs` (179 KB!)

For range and equality predicates on sortable columns: `id >= 100`, `name = 'foo'`, `created_at BETWEEN x AND y`. The **default** general-purpose scalar index. Internally it's a B+-tree-like structure of sorted batches with statistics on each batch.

Open the file, find the public types: `BTreeIndex`, `OrderableScalarValue`, `BatchStats`. You don't read 179 KB; you read the trait impls and the search method to see how a range query lowers to "find the matching batches, then scan them."

#### Bitmap — `lance-index/src/scalar/bitmap.rs` (81 KB)

For low-cardinality categorical columns. For each *value* in the column, store a `RoaringBitmap` of which row addresses have that value. Equality/IN/NOT IN are bitmap unions/intersections. Cheap to query, cheap to combine, but doesn't help on range queries.

#### ZoneMap — `lance-index/src/scalar/zonemap.rs` (100 KB)

Per-page (zone) min/max statistics. **Very lightweight.** Doesn't precisely answer "which rows match" — instead answers "which pages might match" so the scanner can skip irrelevant pages. Composes well with other indices and with the encoding-level statistics.

#### BloomFilter — `lance-index/src/scalar/bloomfilter.rs` (83 KB)

Probabilistic membership test per page. Useful for `column = X` predicates when X is rare. Has false positives but no false negatives. Underlying SBBF (Split-Block Bloom Filter) implementation in `bloomfilter/sbbf.rs` is a port of Parquet's.

#### LabelList — `lance-index/src/scalar/label_list.rs` (21 KB)

For `List<String>` (or any list-of-scalar) columns. Indexes "row R's list contains value V." Used for tag columns ("show me all rows tagged 'cat'").

#### Inverted (full-text) — `lance-index/src/scalar/inverted/` (huge)

This is the full-text search engine. It tokenizes text columns and builds a posting list per token. Supports BM25 scoring, phrase queries, boolean queries, fuzzy match. The biggest file in the *whole repo* is `scalar/inverted/index.rs` at 197 KB.

Submodules:
- `tokenizer.rs` + `tokenizer/` — pluggable tokenizers (English, jieba for Chinese, lindera for Japanese).
- `parser.rs` — query language parser.
- `query.rs` — query AST (MatchQuery, PhraseQuery, BooleanQuery, ...).
- `wand.rs` — the WAND (Weak AND) algorithm used for top-k retrieval (~70 KB).
- `scorer.rs` — BM25.
- `builder.rs` — index construction (~110 KB).
- `iter.rs` — posting list iterators.
- `encoding.rs` — on-disk encoding for posting lists.

The integration with the scanner's `nearest` is via `FullTextSearchQuery` (`lance-index/src/scalar.rs` re-exports). FTS results use the same row-id pipeline as vector search.

#### NGram — `lance-index/src/scalar/ngram.rs` (62 KB)

Substring search (`LIKE '%foo%'`). Builds an n-gram (default trigram) index. Different from the inverted full-text index — n-grams catch substrings that aren't word-aligned.

#### RTree — `lance-index/src/scalar/rtree.rs` (44 KB; gated by `geo` feature)

Spatial index for geometry columns (`ST_Within`, etc.). Built on `geo-traits`/`geoarrow`.

### 2.3 Expression evaluation — `lance-index/src/scalar/expression.rs` (108 KB) (10 min)

Open the file. The crucial entry point is the **`ScalarIndexExpr`** type and the planner integration (`PlannerIndexExt`).

When the scanner gets a `WHERE` clause, it walks the expression tree and tries to find subexpressions that one or more indices can answer. The result is a tree of `IndexExprResult` nodes interleaved with non-index DataFusion expressions. Index nodes return a `RowAddrTreeMap`; non-index nodes operate on materialized data.

The scanner uses these results as a **prefilter mask** before reading row data, dramatically cutting the I/O. This is the heart of index-driven scans.

### 2.4 The on-disk format — `lance-index/src/scalar/lance_format.rs` (5 min)

Skim it. All scalar indices use Lance's own file format (the one you learned on day 3) for their on-disk storage. So an index is just another `.lance` file (or pair of files) under `_indices/<uuid>/`. The `IndexFile` struct in `lance-table::format` knows the `path` and `size_bytes`.

This means the encoding/file layers are also the index storage layer — a unification you wouldn't see in Parquet-based systems.

## Block 3 — Python bindings (45 min)

### 3.1 What PyO3 actually does (10 min)

PyO3 is the bridge: Rust functions annotated with `#[pyfunction]` and types annotated with `#[pyclass]` become callable from Python. The build tool `maturin` produces a Python wheel containing the compiled Rust extension.

The build entry point is [`python/Cargo.toml`](../python/Cargo.toml). Notice it's **not** a member of the workspace (`exclude = ["python", "java/lance-jni"]` in the root `Cargo.toml`) — it's built separately so users can install just the Python package without the rest of the workspace.

### 3.2 The bindings layer — `python/src/lib.rs` (15 min)

Open [`python/src/lib.rs`](../python/src/lib.rs). Read lines 1–150.

Look for the **`#[pymodule]`** function (typically near the bottom). It registers all the classes and functions that Python sees:

- `LanceFileReader`, `LanceFileWriter`
- `LanceDataset`, `Scanner`
- `MergeInsertBuilder`
- Various utility classes for transactions, sessions, mem-WAL, optimization, etc.

Each is implemented in a sibling file:

```
python/src/
├── lib.rs              # the #[pymodule] entry
├── dataset.rs          # LanceDataset (174 KB — biggest binding file)
├── dataset/            # Submodules: blob, optimize, commit, stats, ...
├── scanner.rs          # Scanner
├── file.rs             # File reader/writer
├── fragment.rs         # Fragment access
├── indices.rs          # Index management
├── transaction.rs
├── namespace.rs
├── schema.rs
├── session.rs
└── tracing.rs          # Bridge for tracing -> Python logger
```

### 3.3 The thin-wrapper philosophy (10 min)

The first principle in `python/AGENTS.md` is **"keep bindings as thin wrappers — centralize validation and logic in Rust core."** That means:

- Validation lives in Rust. The Python wrapper does almost no business logic.
- The Python class often holds an `Arc<RustType>` and forwards method calls.
- Async functions are run on a tokio runtime that the bindings own (`python/src/executor.rs`).

This is why `python/src/dataset.rs` is huge — it has one wrapper method per public Lance API. But each wrapper is small and almost mechanical: convert PyArrow inputs to Arrow, call the Rust method, convert errors to `PyErr`.

Open `python/src/scanner.rs` (only ~5 KB). It's a great representative example. Each method is 5-15 lines of "convert args, call core, return result."

### 3.4 The Python-side glue (5 min)

The bindings expose a minimal surface; the user-facing Python API lives in `python/python/lance/*.py`:

```bash
ls python/python/lance/
```

Files include `dataset.py`, `vector.py`, `tags.py`, etc. They wrap the Rust extension with a Pythonic API and add type hints, docstrings, and convenience methods.

This is also where you find `lance.write_dataset`, `lance.dataset`, `LanceDataset.create_index`, etc. — the entry points users actually call.

### 3.5 Testing the bindings (5 min)

Per `python/AGENTS.md`:

```bash
cd python
uv sync --extra tests --extra dev
uv run maturin develop          # build the Rust extension
uv run pytest python/tests/test_dataset.py -k test_create
```

If you have time today, run this once. Watching `maturin develop` rebuild the Rust crate and load it into Python is the clearest demonstration of the binding model.

## Block 4 — Java bindings overview (30 min)

The Java bindings are similar in structure but use **JNI** (the Java Native Interface) instead of PyO3. Lance has a Maven build that wraps a Rust crate `lance-jni`.

Skim the layout:

```bash
ls java/lance-jni/src/
```

Notable files:

- `lib.rs` — JNI entry point, `#[no_mangle]` `Java_<package>_<class>_<method>` functions
- `blocking_dataset.rs` (121 KB) — the dataset binding, but **blocking** (Java threads can't easily live inside tokio's runtime, so the JNI layer drives a tokio runtime per call).
- `blocking_scanner.rs`, `blocking_blob.rs` — same pattern.
- `async_scanner.rs` — exception: when Java needs an async stream, this provides one with explicit poll semantics.
- `transaction.rs`, `merge_insert.rs`, `optimize.rs`, `index.rs` — feature-specific bindings.
- `error.rs` — converts Lance `Error` to Java exceptions.

For day 5 just *know it exists*. Reading the JNI layer in depth is rarely useful unless you're contributing to Java specifically.

The Maven side lives in `java/com/`, `java/dev/`, and `java/src/`. See `java/AGENTS.md` for build and test commands.

You have a working mental model of the entire stack. Here are the directions you can specialize from.

### Path A: Production deployment

If you'll *use* Lance for an AI/ML workload:

1. Read [`skills/lance-user-guide/SKILL.md`](../skills/lance-user-guide/SKILL.md) and `references/index-selection.md` in the same folder. Excellent practical guidance.
