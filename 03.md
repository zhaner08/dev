## 1. Apache Arrow in 10 minutes

[Arrow](https://arrow.apache.org/) is a **columnar in-memory data format**.
The `arrow-rs` family of crates is the official Rust implementation.

The hierarchy you care about:

```
Schema           // collection of named columns + metadata
└── Field        // one named column with a DataType + nullability
    └── DataType // Int32, Utf8, FixedSizeList<Float32>, Struct, ...

RecordBatch      // a "page" of rows: schema + Vec<ArrayRef>
└── ArrayRef = Arc<dyn Array>  // one column's worth of values
    ├── Int32Array
    ├── Float32Array
    ├── StringArray
    ├── FixedSizeListArray   // <-- the vector column type
    ├── ListArray, StructArray, ...
```

A `RecordBatch` is **the unit of data movement** in LanceDB. You don't pass
single rows around. You pass batches.

### Reading the example schema

From `examples/simple.rs`:

```rust
let schema = Arc::new(Schema::new(vec![
    Field::new("id", DataType::Int32, false),
    Field::new(
        "vector",
        DataType::FixedSizeList(
            Arc::new(Field::new("item", DataType::Float32, true)),
            DIM as i32,
        ),
        true,
    ),
]));
```

Translate this English: "this table has two columns: an `id` (`Int32`,
non-null) and a `vector` (a fixed-size list of 128 `Float32`s, the list
itself is nullable, items inside are nullable too)."

The inner `Field` ("item") is **named** by Arrow convention even though we
don't use it. Don't be confused — that's just Arrow's structure.

### Building a `RecordBatch`

```rust
let batch = RecordBatch::try_new(
    schema.clone(),
    vec![
        Arc::new(Int32Array::from_iter_values(0..1000)),
        Arc::new(FixedSizeListArray::from_iter_primitive::<Float32Type, _, _>(
            (0..1000).map(|_| Some(vec![Some(1.0); 128])),
            128,
        )),
    ],
)?;
```

* `Int32Array::from_iter_values` — construct from a closed iterator (no
  nulls). Returns `Int32Array`.
* `FixedSizeListArray::from_iter_primitive::<Float32Type, _, _>` — a
  generic constructor. The first generic param tells Arrow what primitive
  type the list contains.
* `Arc::new(...)` — wrap in `Arc` because `RecordBatch::try_new` takes
  `Vec<ArrayRef>` and `ArrayRef = Arc<dyn Array>`.

### Vector column = `FixedSizeList<Float16/Float32>`

This is the **single most important fact about LanceDB's data model**:

> A column is a **vector column** iff its `DataType` is `FixedSizeList` of
> `Float16` or `Float32`. The `list_size` is the dimension.

When LanceDB sees a column matching that shape, it auto-selects vector
indexes (IVF_PQ by default). When you call `table.vector_search(...)` it
finds vector columns this way.

You'll find this check in many places — most notably `index.rs`:

```rust
// roughly
fn supported_vector_data_type(dt: &DataType) -> bool {
    matches!(dt, DataType::FixedSizeList(field, _)
        if matches!(field.data_type(), DataType::Float16 | DataType::Float32))
}
```

(Real impl is in `rust/lancedb/src/utils/mod.rs` — open it and grep
`supported_vector_data_type`.)

## 2. RecordBatchReader vs RecordBatchStream

Arrow provides `arrow_array::RecordBatchReader`: a synchronous iterator over
batches plus a `schema()`. LanceDB defines its own `RecordBatchReader` and
`RecordBatchStream` in `arrow.rs` for the async case.

Open `rust/lancedb/src/arrow.rs` and look at:

```rust
pub trait RecordBatchReader: Iterator<Item = Result<arrow_array::RecordBatch>> {
    fn schema(&self) -> Arc<arrow_schema::Schema>;
}

pub trait RecordBatchStream: Stream<Item = Result<arrow_array::RecordBatch>> {
    fn schema(&self) -> Arc<arrow_schema::Schema>;
}

pub type SendableRecordBatchStream = Pin<Box<dyn RecordBatchStream + Send>>;
```

When you write data → LanceDB takes a `Box<dyn RecordBatchReader + Send>` (or
something convertible). When you read data → LanceDB returns a
`SendableRecordBatchStream`.

**You usually don't construct readers manually.** Convenient `Into` impls
exist so you can pass:

* A `RecordBatch` directly (LanceDB wraps it in a single-batch reader).
* A `Vec<RecordBatch>`.
* An `arrow_array::RecordBatchIterator`.
* A `Box<dyn RecordBatchReader + Send>` (this is the manual escape hatch).

These conversions live in `data/scannable.rs`. Look at the `Scannable` trait
and its impls.

```rust
// rust/lancedb/src/data/scannable.rs (abbrev.)
pub trait Scannable: Send {
    fn schema(&self) -> SchemaRef;
    fn scan_as_stream(&mut self) -> SendableRecordBatchStream;
    fn num_rows(&self) -> Option<usize> { None }
    fn rescannable(&self) -> bool { false }
}

impl Scannable for RecordBatch { ... }
// also for Vec<RecordBatch>, Box<dyn RecordBatchReader + Send>, etc.
```

The `rescannable` bit is interesting: if your data source is a finite
in-memory table, LanceDB can re-scan it to retry failed writes. If it's a
network stream, it can't.

## 3. The Lance crate family

LanceDB's storage layer is **Lance**. From your reading on Day 1:

| Crate | Star types |
|---|---|
| `lance` | `lance::Dataset`, `WriteParams`, `ReadParams`, `WriteMode`. |
| `lance::dataset::scanner` | `Scanner`, `DatasetRecordBatchStream`. |
| `lance::index` | `DatasetIndexExt`, `vector::VectorIndexParams`. |
| `lance-table::format::Manifest` | The "what version are we on" doc. |
| `lance-table::io::commit` | Custom commit handlers (e.g. for namespaces). |
| `lance-index::scalar::FullTextSearchQuery` | The FTS query type. |
| `lance-index::vector::DIST_COL` | Constant: the column name `"_distance"`. |
| `lance-index::scalar::inverted::SCORE_COL` | Constant: `"_score"`. |
| `lance-io::object_store::*` | `ObjectStoreParams`, `StorageOptionsAccessor`. |
| `lance-linalg::distance::DistanceType` | The L2/Cosine/Dot/Hamming enum. |
| `lance-namespace` / `lance-namespace-impls` | The catalog tree abstraction. |

What "Lance vs LanceDB" responsibilities look like:

| Concern | Owner |
|---|---|
| File format on disk | `lance-file`, `lance-encoding` |
| Manifests, commits, versions | `lance-table` |
| IVF / HNSW / PQ / SQ / FTS indexes | `lance-index` |
| Streaming scan, predicate pushdown | `lance` (Dataset) |
| Object store (S3, GCS, Azure, FS) | `lance-io` + `object_store` |
| Distance metrics | `lance-linalg` |
| Catalog (the *idea* of a database with named tables) | LanceDB |
| Embedding pipeline (auto-embed on insert) | LanceDB |
| Builders / ergonomics / Python+Node bindings | LanceDB |
| Cloud (HTTP) backend | LanceDB |

When you hit a function call into `lance::*` while reading, that's the
boundary — you can mostly stop reading and trust it. (Of course, when
debugging, you'll cross over.)

## 4. `object_store` — multi-backend storage

The [`object_store`](https://docs.rs/object_store) crate is the workhorse for
"talk to S3 / GCS / Azure / local FS uniformly". It provides one trait,
`ObjectStore`, with `get`, `put`, `list`, `delete` etc.

LanceDB doesn't use `object_store` directly that often — it goes through
`lance-io`'s wrappers. But you'll see types like `object_store::path::Path`
and `object_store::Error` in error chains.

When you provide a URI like `s3://bucket/path/to/db`, the chain is:

```
"s3://..." → lancedb::connect()  → ConnectBuilder.execute()
                                  → ListingDatabase
                                  → lance::ObjectStoreParams (via io/object_store.rs)
                                  → lance-io picks AwsObjectStore
                                  → object_store::aws::AmazonS3
```

Open `rust/lancedb/src/io/object_store.rs` (~250 lines) — it's small. It
mainly merges per-table storage options into Lance's `ObjectStoreParams`.

## 5. DataFusion in one paragraph

[DataFusion](https://arrow.apache.org/datafusion/) is an embeddable SQL +
DataFrame execution engine in Rust, built on Arrow. LanceDB uses it for:

* **Logical expressions**: `datafusion_expr::Expr` — used in `Filter`,
  `Select`, `update_columns(...)`, etc. So you can pass a typed expression
  instead of a SQL string.
* **Physical execution plans**: the streaming scan & insert pipeline are
  `datafusion_physical_plan::ExecutionPlan` impls. See
  `table/datafusion/insert.rs`.
* **SQL parsing**: when you write `table.update().only_if("id > 10")`, that
  string gets parsed by `datafusion-sql`.

You don't need to be a DataFusion expert. Whenever you see `Expr`,
`ExecutionPlan`, `TaskContext`, or `SessionContext`, think "DataFusion".

The relevant LanceDB files are `expr.rs`, `expr/sql.rs`, and
`table/datafusion/`.

## 6. Putting it together — a tiny scenario

Consider:
```rust
let db = connect("s3://my-bucket/lancedb").execute().await?;
let tbl = db.open_table("docs").execute().await?;
let stream = tbl.query()
    .only_if("score > 0.5")
    .select(Select::Columns(vec!["id".into(), "vector".into()]))
    .nearest_to(&[0.1; 128])?
    .execute()
    .await?;
let batches: Vec<RecordBatch> = stream.try_collect().await?;
```

What types pass through:

1. `connect` returns a `ConnectBuilder`.
2. `.execute().await?` returns a `Connection`. Internally it builds a
   `ListingDatabase` (because URI starts with `s3://`).
3. `ListingDatabase` uses `object_store::aws::AmazonS3` under the hood.
4. `db.open_table("docs").execute()` returns a `Table` wrapping a
   `Arc<dyn BaseTable>` which is a `NativeTable` referencing a
   `lance::Dataset`.
5. `tbl.query()` returns a `Query`. `only_if("score > 0.5")` parses the
   string into an `Expr` via DataFusion. `select(...)` records a `Select::
   Columns`. `nearest_to([..])` upgrades it to `VectorQuery`.
6. `.execute().await?` calls `BaseTable::create_plan` → a
   `datafusion_physical_plan::ExecutionPlan` → `lance_datafusion::exec::
   execute_plan` → `SendableRecordBatchStream`.
7. `try_collect` drains the stream into a `Vec<RecordBatch>`.

All of this you'll meet in detail on Days 2 PM and 3.

## 1. The user's view

```rust
let db = lancedb::connect("data/sample-lancedb").execute().await?;
db.create_table("docs", batch).execute().await?;
db.open_table("docs").execute().await?;
db.table_names().execute().await?;
db.drop_table("docs", &[]).await?;
```

Three things to notice:

* `connect` is a *free function* that returns a builder — not the connection
  itself.
* `Connection::create_table` and `open_table` also return builders. Each
  builder ends in `.execute().await?`.
* The methods on `Connection` look thin. The interesting work is in the
  underlying `Database` trait object.

## 2. `connect` and `ConnectBuilder`

Open `rust/lancedb/src/connection.rs`. Search for `pub fn connect` (it's near
the bottom, around line 850):

```rust
/// Connect to a LanceDB database.
pub fn connect(uri: &str) -> ConnectBuilder {
    ConnectBuilder::new(uri)
}
```

`ConnectBuilder` (defined a few hundred lines up) holds:

```rust
struct ConnectRequest {
    uri: String,
    options: HashMap<String, String>,        // storage options (S3 keys, etc.)
    read_consistency_interval: Option<Duration>,
    session: Option<Arc<lance::session::Session>>,
    client_config: Option<ClientConfig>,     // remote-only
    manifest_enabled: bool,                  // local "manifest mode"
    namespace_client_properties: HashMap<String, String>,
}
```

`ConnectBuilder` itself wraps that plus an optional `embedding_registry` and
exposes methods:

```rust
.storage_option(k, v)
.storage_options([(k, v), ...])
.read_consistency_interval(Duration::from_secs(5))
.api_key("...")              // remote
.region("us-east-1")         // remote
.host_override("...")        // remote
.session(custom_session)
.execute()                   // <-- the terminal
```

The terminal `execute()` looks like (search for `pub async fn execute(self)
-> Result<Connection>`):

```rust
pub async fn execute(self) -> Result<Connection> {
    if self.request.uri.starts_with("db") {
        self.execute_remote()                                    // db:// URI
    } else if self.request.manifest_enabled {
        let internal = Arc::new(
            ListingDatabase::connect_manifest_enabled_namespace_database(&self.request).await?
        );
        Ok(Connection { internal, embedding_registry: ... })
    } else {
        let internal = Arc::new(ListingDatabase::connect_with_options(&self.request).await?);
        Ok(Connection { internal, embedding_registry: ... })
    }
}
```

Three branches, three implementations of `Database`:

| URI prefix             | Database impl                       |
|------------------------|-------------------------------------|
| `db://...`             | `RemoteDatabase` (Cloud)            |
| any, `manifest_enabled = true` | `LanceNamespaceDatabase` over a directory namespace |
| any (default)          | `ListingDatabase`                   |

`Connection` itself is just:

```rust
pub struct Connection {
    internal: Arc<dyn Database>,
    embedding_registry: Arc<dyn EmbeddingRegistry>,
}
```

A handle to a `Database` plus an embedding registry. It implements
`Clone` (via `Arc::clone` of the two fields), so passing `Connection`s around
is cheap.

## 3. The `Database` trait — the catalog interface

Open `rust/lancedb/src/database.rs`. Read all of it (~250 lines).

The trait:

```rust
#[async_trait::async_trait]
pub trait Database: Send + Sync + std::any::Any + Debug + Display + 'static {
    fn uri(&self) -> &str;
    async fn read_consistency(&self) -> Result<ReadConsistency>;

    async fn list_namespaces(&self, ...) -> Result<...>;
    async fn create_namespace(&self, ...) -> Result<...>;
    async fn drop_namespace(&self, ...) -> Result<...>;
    async fn describe_namespace(&self, ...) -> Result<...>;

    async fn list_tables(&self, ...) -> Result<ListTablesResponse>;

    async fn create_table(&self, request: CreateTableRequest)  -> Result<Arc<dyn BaseTable>>;
    async fn clone_table (&self, request: CloneTableRequest)   -> Result<Arc<dyn BaseTable>>;
    async fn open_table  (&self, request: OpenTableRequest)    -> Result<Arc<dyn BaseTable>>;

    async fn rename_table(&self, ...) -> Result<()>;
    async fn drop_table  (&self, name: &str, ns: &[String]) -> Result<()>;
    async fn drop_all_tables(&self, ns: &[String]) -> Result<()>;

    async fn namespace_client(&self) -> Result<Arc<dyn LanceNamespace>>;
    async fn namespace_client_config(&self) -> Result<(String, HashMap<String, String>)>;
    fn as_any(&self) -> &dyn std::any::Any;
}
```

Key observations:

* **Every meaningful method takes a `Request` struct, not a positional
  argument list.** That's how new fields get added without breaking
  existing impls. Look at `OpenTableRequest`, `CreateTableRequest`,
  `CloneTableRequest`, `TableNamesRequest`, `ListTablesRequest`. They're all
  in this file (or imported from `lance_namespace::models`).
* **Returned tables are `Arc<dyn BaseTable>`** — same trait-object trick
  the `Database` itself uses, applied one level down.
* **Namespaces are first-class**. `Database` doesn't just track tables, it
  tracks a tree of namespaces (a single root by default). Most casual usage
  doesn't touch this; advanced multi-tenant usage does.

`ReadConsistency` enum (also in this file) controls how stale reads can be:

* `Manual` — only refresh when explicitly asked.
* `Eventual(Duration)` — refresh after this many seconds since last check.
* `Strong` — refresh on every read.

This propagates down into `DatasetConsistencyWrapper` in
`table/dataset.rs` (we'll meet it on Day 3).

## 4. `ListingDatabase` — the local implementation

Open `rust/lancedb/src/database/listing.rs`. It's ~95 KB; we'll skim, not
read all.

The high-level shape:

```rust
pub struct ListingDatabase {
    object_store: Arc<ObjectStore>,            // from lance-io
    base_path: object_store::path::Path,
    uri: String,
    read_consistency_interval: Option<Duration>,
    storage_options: HashMap<String, String>,
    new_table_config: NewTableConfig,
    session: Arc<lance::session::Session>,
    // ... and various commit-handler / namespace-aware fields
}
```

`ListingDatabase::connect_with_options` does roughly:

1. Parse the URI (`file://` ⇢ local fs, `s3://` ⇢ S3, etc.) using
   `lance::io::ObjectStore::from_uri_and_params`. This delegates to the
   `object_store` crate via `lance-io`.
2. Resolve / create the base path (`create_dir_all` for local).
3. Build a Lance `Session` if one wasn't provided.
4. Stash storage options for child tables.

When you later call `database.open_table(request)`, `ListingDatabase`:

1. Validates the table name (no slashes etc.) — `validate_table_name`.
2. Constructs the table directory path: `<base>/{namespace_path}/{name}.lance`.
3. Calls `lance::dataset::builder::DatasetBuilder::from_uri(...)
   .with_object_store(...).with_read_params(...)` to materialize a
   `lance::Dataset`.
4. Wraps it in a `NativeTable`, returns `Arc<dyn BaseTable>`.

`create_table`:

1. Validates the table name.
2. Streams the input data through the embedding pipeline (if any embeddings
   are defined) — see `data/scannable.rs` `compute_embeddings_for_batch`.
3. Calls `lance::dataset::write(...)` (via `WriteParams`), which produces a
   new dataset.
4. Returns the resulting `NativeTable`.

`table_names` / `list_tables` lists files in the base directory whose name
ends with `.lance`. (Hence the name "listing" database.)

The takeaway: **local LanceDB is just a directory and a naming convention.**
There is no central "catalog file" — the directory listing is the catalog.
This is similar to early Hive metastore-less Parquet warehouses.

## 5. `LanceNamespaceDatabase` — namespace-managed

Open `rust/lancedb/src/database/namespace.rs`.

The difference vs `ListingDatabase`:

* Backed by an explicit catalog (a `LanceNamespace` impl from
  `lance-namespace-impls`) — could be a directory namespace, a REST
  namespace, or a custom Postgres/DynamoDB-backed one (those live in
  external impls).
* Tables are not discovered by listing files; they're discovered by asking
  the namespace.
* Manifests can be managed centrally instead of per-directory. This enables
  features like atomic multi-table transactions (in some namespace impls).

For most users (especially while learning), this is identical to
`ListingDatabase`. You only switch to `LanceNamespaceDatabase` when you opt
into `manifest_enabled(true)` on the connection, or use a `db://` URI to
talk to a service.

Skim this file's `impl Database for LanceNamespaceDatabase` block. Notice
that most methods boil down to "delegate to the inner `LanceNamespace`
client + maybe load a `lance::Dataset` to wrap as `NativeTable`".

## 7. How a `create_table` actually flows

Let's trace `db.create_table("docs", batch).execute().await?`:

1. **`Connection::create_table`** (in `connection.rs`):
   ```rust
   pub fn create_table<T: Scannable + 'static>(
       &self, name: impl Into<String>, initial_data: T
   ) -> CreateTableBuilder {
       let initial_data = Box::new(initial_data);
       CreateTableBuilder::new(self.internal.clone(), self.embedding_registry.clone(),
                               name.into(), initial_data)
   }
   ```
   It boxes the data behind a `Box<dyn Scannable>`. The generic `T:
   Scannable + 'static` is what lets you pass a `RecordBatch`, a
   `Vec<RecordBatch>`, a `Box<dyn RecordBatchReader + Send>`, or anything
   that impls `Scannable`. (See Day 2 AM.)

2. **`CreateTableBuilder` chain** (in `connection/create_table.rs`):
   - `.mode(...)`, `.add_embedding(...)`, `.write_options(...)`, etc.
     accumulate state.
   - `.execute()` builds a `CreateTableRequest` and calls
     `self.parent.create_table(request).await`.

3. **`Database::create_table`**:
   - `ListingDatabase::create_table` validates the name, resolves a path,
     coerces input batches, writes a new Lance dataset via
     `lance::dataset::write`, returns a `NativeTable`.
   - `RemoteDatabase::create_table` instead uploads the data as Arrow IPC
     in an HTTP POST.

4. **The builder's `execute` wraps the result in `Table::new(...)`** to give
   you back a `lancedb::Table` (the public newtype around `Arc<dyn
   BaseTable>`). That's what your code holds onto.

This pattern — *Connection method returns Builder; Builder accumulates a
Request struct; Request goes through a trait-object Database; trait-object
returns a trait-object Table* — repeats for every operation. Internalize it.

## 2. The `Table` struct

```rust
// rust/lancedb/src/table.rs (~ line 371)
pub struct Table {
    inner: Arc<dyn BaseTable>,
    database: Option<Arc<dyn Database>>,    // back-ref, used by some ops
    embedding_registry: Arc<dyn EmbeddingRegistry>,
}
```

`Table` is **`Clone`** (cheap — just three Arc bumps). It implements `Display`
and `Debug`. Its public methods fall into three buckets:

* **Trivial delegations** that just call `self.inner.<method>().await` and
  forward the result. Examples: `name()`, `schema()`, `count_rows()`,
  `version()`, `list_indices()`, `delete()`.
* **Builder factories** that construct a builder, pre-fill in
  `Arc::clone(&self.inner)`, and return it for the user to configure:
  ```rust
  pub fn add<T: Scannable + 'static>(&self, data: T) -> AddDataBuilder {
      AddDataBuilder::new(
          self.inner.clone(),
          Box::new(data),
          Some(self.embedding_registry.clone()),
      )
  }
  pub fn update(&self) -> UpdateBuilder { UpdateBuilder::new(self.inner.clone()) }
  pub fn merge_insert(&self, on: &[&str]) -> MergeInsertBuilder { ... }
  pub fn create_index(&self, columns: &[&str], idx: Index) -> IndexBuilder { ... }
  pub fn query(&self) -> Query { Query::new(self.inner.clone()) }
  pub fn vector_search<V: IntoQueryVector>(&self, q: V) -> Result<VectorQuery> { ... }
  ```
* **A few helpers that combine multiple base methods** — e.g. opening
  a snapshot at a tag.

You'll note `Table::query()` and `Table::vector_search()` build query objects
without going through `BaseTable` immediately. The `BaseTable` operations
(`create_plan`, `query`) only get hit when the user finally calls
`.execute()` on the query builder. We meet `Query` on Day 3 PM.

## 4. `NativeTable`

```rust
// rust/lancedb/src/table.rs (~ line 1261)
pub struct NativeTable {
    name: String,
    namespace: Vec<String>,
    id: String,
    uri: String,
    pub(crate) dataset: dataset::DatasetConsistencyWrapper,    // wraps Arc<lance::Dataset>
    read_consistency_interval: Option<Duration>,
    pub(crate) namespace_client: Option<Arc<dyn LanceNamespace>>,
    pub(crate) pushdown_operations: HashSet<NamespaceClientPushdownOperation>,
}
```

The interesting field is `dataset: DatasetConsistencyWrapper` (defined in
`table/dataset.rs`). Open that file.

```rust
pub struct DatasetConsistencyWrapper {
    state: Arc<Mutex<DatasetState>>,        // current Arc<lance::Dataset>
    consistency: ConsistencyMode,           // Lazy / Strong / Eventual(BackgroundCache)
}
```

Three consistency modes, mapped from the `read_consistency_interval`:

* `None` → `Lazy`: only refresh on explicit `checkout_latest()`.
* `Some(0)` → `Strong`: re-check the manifest version on every read.
* `Some(d)` → `Eventual(BackgroundCache)`: if the dataset is older than `d`,
  refresh; otherwise reuse the cached one. While idle, the cache refreshes
  in the background.

`BackgroundCache` lives in `utils/background_cache.rs` (you peeked at it on
Day 1). It's a custom-built cache that reuses `futures::future::Shared` so
that concurrent callers all wait on the same in-flight refresh future.
(The `moka` workspace dep is used elsewhere — for the index cache, not
here.)

How the trait maps onto Lance:

| `BaseTable` method | What `NativeTable` does |
|---|---|
| `schema` | `dataset.get().await?.schema()`, return as `SchemaRef`. |
| `count_rows` | `dataset.scanner()` → push down filter → call `count_rows`. |
| `add` | Build a DataFusion `ScannableExec` over the input, call `lance::dataset::InsertBuilder` (writes a new fragment, commits). |
| `delete` | `dataset.delete(predicate)` (Lance writes a deletion vector). |
| `update` | Build an `UpdateJob` (Lance) from the predicate + `SET` exprs. |
| `merge_insert` | `dataset.merge_insert(...)`, takes `RecordBatchReader`. |
| `create_index` | `lance::index::DatasetIndexExt::create_index(params)`. |
| `optimize` | Compaction / index optimization via `lance::dataset::optimize`. |
| `query` / `create_plan` | Build a `lance::dataset::scanner::Scanner`, attach query options, return its plan / stream. (Most of this is in `table/query.rs`.) |

Whenever Lance returns a *new dataset version* (because a write happened),
`NativeTable` calls `self.dataset.update(new_dataset)` to refresh the cached
`Arc<lance::Dataset>`.

## 5. The mutation builders — bird's-eye

Each mutation has a small builder file under `rust/lancedb/src/table/`. They
all follow the same structure: a public builder struct, configuration
methods, a terminal `execute()` (or `await`) that calls into `BaseTable`.

### `AddDataBuilder` (file: `table/add_data.rs`)

```rust
pub struct AddDataBuilder {
    pub(crate) parent: Arc<dyn BaseTable>,
    pub(crate) data: Box<dyn Scannable>,
    pub(crate) mode: AddDataMode,            // Append (default) | Overwrite
    pub(crate) write_options: WriteOptions,  // wraps lance WriteParams
    pub(crate) on_nan_vectors: NaNVectorBehavior,  // Error | Keep
    pub(crate) embedding_registry: Option<Arc<dyn EmbeddingRegistry>>,
    pub(crate) progress_callback: Option<ProgressCallback>,
    pub(crate) write_parallelism: Option<usize>,
}
```

Notable methods:
* `.mode(AddDataMode::Overwrite)` to overwrite instead of append.
* `.add_embedding(EmbeddingDefinition)` to compute embeddings on insert.
* `.write_progress(...)` for progress callbacks.
* `.execute()` calls `self.parent.add(self).await`.

### `UpdateBuilder` (file: `table/update.rs`)

```rust
table.update()
    .only_if("score > 0.5")
    .column("flag", "true")               // expr
    .execute().await?;
```

Either set columns by name with SQL/exprs, or replace whole columns. The
`only_if` filter is parsed by DataFusion. Read the file top to bottom — it's
short.

### Delete (file: `table/delete.rs`)

Delete is so simple it doesn't need a builder:
```rust
table.delete("id > 100").await?;
```
Returns a `DeleteResult { version }`.

### `MergeInsertBuilder` (file: `table/merge.rs`)

This is **UPSERT** (or "MERGE INTO"). You build it with the columns to
match on, then optionally configure what happens on match / no-match:

```rust
let mut builder = table.merge_insert(&["id"]);
builder.when_matched_update_all(None)
       .when_not_matched_insert_all();
let result = builder.execute(reader).await?;
```

The trait method `merge_insert(&self, params, new_data: Box<dyn
RecordBatchReader + Send>)` lives on `BaseTable`. The builder packages the
config; the trait does the work.

### Schema evolution (`table/schema_evolution.rs`)

`add_columns`, `alter_columns`, `drop_columns`. Add columns can derive new
column values from existing ones via a `NewColumnTransform` (re-exported
from Lance). Useful for adding an embedding column over an existing text
column.

### `optimize` (`table/optimize.rs`)

`OptimizeAction` enum:
* `Compact` — merge small fragments.
* `IndexCleanUp` — delete old index versions.
* `OptimizeIndices` — rebuild deltas into the main index.
* `All` — all of the above.

Returns `OptimizeStats`. This is what you call when a table has had a lot of
churn and is getting slow.

# Day 3, Afternoon — Query and Index

Time: ~2 hours. Goal: understand how queries are built, executed, and how
indexes back them. After this chapter, you can read `query.rs`, `index.rs`,
`index/vector.rs`, `index/scalar.rs`, and `query/hybrid.rs` confidently.

## 1. Three kinds of query, one execution model

A LanceDB query is one of:

* **Plain**: scan the table, optionally filter, optionally select. Returns
  rows in arbitrary order.
* **Vector**: nearest-neighbor search by vector similarity. Returns rows in
  *increasing distance* order, with a `_distance` column.
* **Full-text (FTS)**: BM25 search over a text column. Returns rows in
  *decreasing score* order, with a `_score` column.
* **Hybrid**: vector + FTS combined and reranked.

All are built fluently:

```rust
// plain
table.query().only_if("score > 0.5").select(Select::Columns(vec!["id".into()]))
     .limit(100).execute().await?;

// vector
table.vector_search(&[0.1; 128])?.distance_type(DistanceType::Cosine)
     .nprobes(30).limit(15).execute().await?;

// fts
table.query().full_text_search(FullTextSearchQuery::new("dog cat".into()))
     .limit(10).execute().await?;

// hybrid
table.query().full_text_search(FullTextSearchQuery::new("dog".into()))
     .nearest_to(&query_vector)?.limit(5)
     .execute_hybrid(QueryExecutionOptions::default()).await?;
```

All four ultimately produce a `SendableRecordBatchStream`. Open
`rust/lancedb/src/query.rs` and follow along.

---

## 2. The trait family in `query.rs`

```rust
pub trait QueryBase {                                  // line 374
    fn limit(self, ...) -> Self;
    fn offset(self, ...) -> Self;
    fn only_if(self, filter: impl AsRef<str>) -> Self;
    fn only_if_expr(self, filter: datafusion_expr::Expr) -> Self;
    fn full_text_search(self, q: FullTextSearchQuery) -> Self;
    fn select(self, select: Select) -> Self;
    fn fast_search(self) -> Self;
    fn with_row_id(self) -> Self;
    fn postfilter(self) -> Self;
    fn rerank(self, reranker: Arc<dyn Reranker>) -> Self;
    fn norm(self, norm: NormalizeMethod) -> Self;
    ...
}

pub trait HasQuery {                                   // line 515
    fn mut_query(&mut self) -> &mut QueryRequest;
}

impl<T: HasQuery> QueryBase for T { ... }              // blanket impl

pub trait ExecutableQuery {                            // line 621
    fn create_plan(&self, opts) -> impl Future<Output = ...>;
    fn execute(&self) -> impl Future<Output = ...>;
    fn execute_with_options(&self, opts) -> impl Future<Output = ...>;
    fn explain_plan(&self, verbose: bool) -> impl Future<Output = ...>;
    fn analyze_plan(&self, opts) -> impl Future<Output = ...>;
}
```

Why two traits and a blanket impl?

* `QueryBase` exposes the **fluent setter methods** that all query types
  share (limit, filter, select).
* `HasQuery` exposes the *implementation hook*: any struct that contains a
  mutable `QueryRequest` can be a `QueryBase` for free via the blanket impl.
* `ExecutableQuery` exposes the **terminal methods** (`execute`,
  `create_plan`, `explain_plan`).

`Query`, `VectorQuery`, and `TakeQuery` (a special "take rows by ID" path)
all implement `HasQuery` and `ExecutableQuery`, so they share configuration
methods but each implements `execute` differently.

## 4. `Select` — column projection

```rust
pub enum Select {
    All,                                                 // every non-system col
    Columns(Vec<String>),                                // by name
    Dynamic(Vec<(String, String)>),                      // (name, sql_expr)
    Expr(Vec<(String, datafusion_expr::Expr)>),          // typed exprs
}
```

`Dynamic` and `Expr` enable computed columns:


Or with type-safe expressions (preferred when you have them):

```rust
use lancedb::expr::{col, lit};

    ("id2", col("id") * lit(2)),

```

`Expr` is serialized to SQL when sent to a remote table.

## 6. `ExecutableQuery::execute` — what actually runs

For `VectorQuery`:

```rust
impl ExecutableQuery for VectorQuery {
    async fn create_plan(&self, opts) -> Result<Arc<dyn ExecutionPlan>> {
        // 1. Build an AnyQuery::VectorQuery from self.request
        // 2. Call self.parent.create_plan(&query, opts).await  (BaseTable trait)
        // 3. Return the plan
    }
    async fn execute_with_options(&self, opts) -> Result<SendableRecordBatchStream> {
        let plan = self.create_plan(opts).await?;
        // execute_plan from lance-datafusion to drive the plan to a stream
    }
}
```

So **the query traits themselves don't know whether they're hitting a local
or a remote table**. They build a `Query`/`VectorQuery` request, hand it
to `BaseTable::create_plan`, and stream the result.

`BaseTable::create_plan` is implemented:
* In `NativeTable` — see `rust/lancedb/src/table/query.rs`. It builds a
  `lance::dataset::scanner::Scanner`, attaches all the parameters (filter,
  vector, nprobes, refine_factor, etc.), and returns the scanner's
  `ExecutionPlan` adapter.
* In `RemoteTable` — see `rust/lancedb/src/remote/table.rs`. It serializes
  the `QueryRequest` to JSON, POSTs it to `/v1/table/{name}/query/`,
  receives an Arrow IPC stream, and wraps it as an `ExecutionPlan` (or in
  more recent versions, just a `SendableRecordBatchStream`).

## 7. Hybrid search — `query/hybrid.rs`

Hybrid search runs *both* a vector query and an FTS query, then reranks
the merged results. Open `rust/lancedb/src/query/hybrid.rs` (~330 lines) and
read it; it's instructive Rust.

The flow:

1. From a `Query` that has both `query_vector` and `full_text_search` set,
   `Query::execute_hybrid` calls into `hybrid::execute_hybrid_query`.
2. `execute_hybrid_query` runs both queries in parallel using `tokio::try_
   join!` (or `futures::try_join!`).
3. It collects each side's results to a `Vec<RecordBatch>`.
4. It calls a `Reranker` (default: `RRFReranker`) to merge / reorder.
5. Returns the merged batches as a stream.

`RRF` = Reciprocal Rank Fusion (`rerankers/rrf.rs`):
> For each result row, score = sum over each list of `1 / (k + rank_in_list)`.
> Default `k = 60`.

You can plug in custom rerankers. The Python wrapper has many more
(`python/python/lancedb/rerankers/`).

## 8. The index layer — `index.rs`

Open `rust/lancedb/src/index.rs`. The top-level public type is the `Index`
enum:

```rust
pub enum Index {
    Auto,
    BTree(BTreeIndexBuilder),
    Bitmap(BitmapIndexBuilder),
    LabelList(LabelListIndexBuilder),
    FTS(FtsIndexBuilder),
    IvfFlat(IvfFlatIndexBuilder),
    IvfPq(IvfPqIndexBuilder),
    IvfSq(IvfSqIndexBuilder),
    IvfRq(IvfRqIndexBuilder),
    IvfHnswPq(IvfHnswPqIndexBuilder),
    IvfHnswSq(IvfHnswSqIndexBuilder),
    IvfHnswFlat(IvfHnswFlatIndexBuilder),
}
```

Each variant carries a *builder* with index-specific parameters.

Two file groups:

* `rust/lancedb/src/index/scalar.rs` — `BTreeIndexBuilder`, `BitmapIndexBuilder`,
  `LabelListIndexBuilder`, `FtsIndexBuilder`. Read all (~80 lines).
* `rust/lancedb/src/index/vector.rs` — all the vector index builders
  (~470 lines). Read top to bottom; each builder is small.

`Index::Auto` is the friendliest entry point: when you call
`table.create_index(&["vector"], Index::Auto)`, LanceDB inspects the column
type:

* `FixedSizeList<Float16/Float32>` → IVF_PQ with sane defaults.
* `Utf8`/`LargeUtf8` → BTree (or FTS if you say so explicitly).
* numeric → BTree.
* Otherwise → error.

Look at `IndexBuilder` (the wrapping struct that drives the create) and
where it dispatches. Search for `auto_select` in `index.rs`.

### IVF_PQ at a glance

IVF (Inverted File): partition the vector space into Voronoi cells using
k-means. To search, find the nearest cells (= "probes") and only look at
their members. **`nprobes`** controls how many cells to inspect; higher =
slower + more accurate.

PQ (Product Quantization): split each vector into M sub-vectors, train a
small codebook (e.g. 256 entries) per sub-vector, replace each sub-vector
by its codebook index. Compression factor depends on M.

`IvfPqIndexBuilder` (in `index/vector.rs`) lets you configure
`distance_type`, `num_partitions`, `num_sub_vectors`, `sample_rate`, etc.

### HNSW

HNSW = Hierarchical Navigable Small World — graph-based, very fast queries,
not as good as IVF for billion-scale. The IVF_HNSW_* variants combine both:
IVF for coarse, HNSW for fine. Mostly use these for huge datasets.

### Other variants

* `IvfFlat` — no compression, highest recall, biggest index.
* `IvfSq` — Scalar Quantization (8-bit per dim).
* `IvfRq` — RaBitQ Quantization (research-y, very compact).

You don't need to memorize the algorithms. Know **what the names mean** so
you can read configuration code intelligently.

## 1. `EmbeddingFunction` and the registry

Open `rust/lancedb/src/embeddings.rs` (~470 lines). Read all of it.

```rust
pub trait EmbeddingFunction: Debug + Send + Sync {
    fn name(&self) -> &str;
    fn source_type(&self) -> Result<Cow<'_, DataType>>;        // input column type
    fn dest_type(&self)   -> Result<Cow<'_, DataType>>;        // embedding column type
    fn compute_source_embeddings(&self, source: Arc<dyn Array>) -> Result<Arc<dyn Array>>;
    fn compute_query_embeddings(&self, input:  Arc<dyn Array>) -> Result<Arc<dyn Array>>;
}
```

Two methods do the work:
* `compute_source_embeddings` — applied to a column of stored data when
  inserting/updating the table.
* `compute_query_embeddings` — applied to a query value at search time so
  that you can call `table.vector_search("hello")` instead of hand-coding
  the embedding vector.

Why split them? Some models tokenize differently for documents vs queries
(e.g. asymmetric retrieval models). Some return different shapes. The trait
lets the impl choose.

The **registry** is a singleton (per-Connection) where you register
implementations by name:

```rust
pub trait EmbeddingRegistry: Send + Sync + Debug {
    fn functions(&self) -> HashSet<String>;
    fn register(&self, name: &str, function: Arc<dyn EmbeddingFunction>) -> Result<()>;
    fn get(&self, name: &str) -> Option<Arc<dyn EmbeddingFunction>>;
}

pub struct MemoryRegistry { /* RwLock<HashMap<String, Arc<dyn EmbeddingFunction>>> */ }
```

Default registry is `MemoryRegistry` — a thread-safe in-memory map. Custom
registries could load from a config file or fetch from a service.

## 2. `EmbeddingDefinition` + the write pipeline

```rust
pub struct EmbeddingDefinition {
    pub source_column: String,
    pub dest_column: Option<String>,         // default: "{src}_embedding"
    pub embedding_name: String,              // looked up in the registry
}
```

How it gets used:

1. When you call `db.create_table(...).add_embedding(EmbeddingDefinition::
   new("text", "openai", Some("vector")))`, the table's schema gains a
   new column (`vector: FixedSizeList<Float32>(N)`) where N comes from the
   embedding function's `dest_type()`.
2. The column definition stored in metadata becomes
   `ColumnKind::Embedding(definition)` (see `table.rs` line ~99).
3. When you later `table.add(text_only_batch)`, the write pipeline:
   * Reads the table's `TableDefinition`.
   * Wraps the input in `WithEmbeddings`, a `RecordBatchReader` that, for
     each batch, calls `compute_source_embeddings` on the source column and
     appends the result as a new column.
4. The augmented batch (with embedding column) is what actually hits Lance.

Open `embeddings.rs` and find:

* `pub struct WithEmbeddings<R: RecordBatchReader>` — the wrapping reader.
* `pub enum MaybeEmbedded<R: ...>` — `Yes(WithEmbeddings)` or `No(R)`.
* `compute_embeddings_for_batch` — the per-batch worker. Uses
  `std::thread::scope` to compute multiple embedding columns in parallel.
* `compute_output_schema` — extends the table schema with the dest field.

The `data/scannable.rs` module has a small helper
`scannable_with_embeddings(...)` that wraps a `Box<dyn Scannable>` with
`WithEmbeddings` if needed.

This is the **clearest example of "data flowing through the codebase"** in
the Rust layer. Trace it once and it'll click.

## 3. The three Rust embedding implementations

Each lives behind a cargo feature.

### `embeddings/openai.rs` — feature `openai`

* Uses [`async-openai`](https://docs.rs/async-openai) to call the OpenAI
  embeddings API.
* Configured with model name (e.g. `text-embedding-3-small`), API key, and
  optionally a base URL (so you can point it at a compatible endpoint like
  Azure or a local proxy).
* `source_type` = `Utf8`, `dest_type` = `FixedSizeList<Float32>(1536)`
  (default; depends on model).
* Network calls are wrapped in `tokio::runtime::Handle::current().block_on(...)`
  inside `compute_source_embeddings` because the trait is sync.

(That last point is interesting: `EmbeddingFunction` is sync because it gets
called from a `RecordBatchReader::next` chain, which is sync. The HTTP
calls are still async under the hood, just driven inline.)

### `embeddings/bedrock.rs` — feature `bedrock`

* Uses `aws-sdk-bedrockruntime` to call AWS Bedrock embedding models
  (Titan, Cohere, ...).
* Same shape as OpenAI: name, model id, source/dest types.

### `embeddings/sentence_transformers.rs` — feature `sentence-transformers`

* Runs the model **locally** using:
    * `hf-hub` to download model files from HuggingFace.
    * `tokenizers` for tokenization.
    * `candle-core`, `candle-nn`, `candle-transformers` for the actual
      neural net.
* No network at runtime (after first download). Default model:
  `sentence-transformers/all-MiniLM-L6-v2`, 384-dim.
* Look at the file: about 500 lines, but most of that is wiring up tokenizer
  + model + pooling. Read the `EmbeddingFunction` impl and `builder()`.

The Python side (`python/python/lancedb/embeddings/`) has many more
implementations: Cohere, VoyageAI, Jina, Gemini, Watsonx, etc. They live in
Python because the Python ML ecosystem is so much richer; they call HTTP
APIs or use Python ML libraries directly.

## 5. `RemoteTable` — `BaseTable` impl over HTTP

Open `rust/lancedb/src/remote/table.rs`. It's *huge* (~217 KB) but every
method has the same skeleton:

```rust
async fn add(&self, builder: AddDataBuilder) -> Result<AddResult> {
    let body = encode_arrow_ipc(builder.data, ...)?;     // serialize to IPC
    let resp = self.client.post(format!("{path}/add/"))
        .body(body)
        .send_with_retry().await?;
    let result: AddResult = resp.json().await?;          // parse JSON response
    Ok(result)
}
```

Patterns to look for as you skim:

* **Construct URL** by combining `base_url + table_path + endpoint`.
* **Encode request** as either Arrow IPC bytes (for data) or JSON
  (for queries / metadata).
* **POST with retry** via the `RetryClient` wrapper.
* **Decode response** depending on the endpoint:
  * Arrow IPC stream for query results / scan results.
  * JSON for metadata / mutation results.

For example, query execution:
```
table.query()...execute()
  → BaseTable::query (called through ExecutableQuery)
    → RemoteTable::query
      → POST /v1/table/{name}/query/  (JSON body: serialized QueryRequest)
        ← Arrow IPC response stream
      → wrapped in SendableRecordBatchStream and returned
```

`ipc.rs` (~80 lines) is the encode/decode helper.

`remote/util.rs` has small helpers for URL encoding and error mapping.

## 6. The HTTP client and retry policy

Open `rust/lancedb/src/remote/client.rs` (~1100 lines):

```rust

    pub timeout_config: TimeoutConfig,
    pub retry_config:   RetryConfig,
```

The `RemoteClient` struct (look for it) holds a `reqwest::Client`, the

* User agent.
* Extra static headers from config.
* Dynamic headers from `HeaderProvider` (so you can implement OAuth
  refresh).
* `x-request-id` header (random UUID per request, included in error

### Retry — `remote/retry.rs`

Three independent failure budgets (read top of `retry.rs`):

```rust
pub struct RetryCounter<'a> {
    pub request_failures: u8,    // 5xx, transient HTTP errors
    pub connect_failures: u8,    // TCP-level failures
    pub read_failures:    u8,    // body decode failures
    pub config: &'a ResolvedRetryConfig,
    pub request_id: String,
}
```

Each failure category has its own max. `increment_from_error` looks at the
underlying `reqwest::Error` and decides which counter to bump. When any
counter exhausts, the wrapped operation fails with `Error::Retry { ...
all_three_counters ... }`. That error carries enough info for users to
debug.

Backoff: exponential with jitter, configurable in `RetryConfig`.

The structure here is a great study case for "how to add structured retries
to an async HTTP client". Read `retry.rs` carefully.

## 7. Putting it together — the full call

Here's what `table.add(batch).execute().await?` looks like end-to-end on
the **remote** side:

```
Table::add(batch)                                         (table.rs)
  → returns AddDataBuilder
.execute()                                                (table/add_data.rs)
  → AddDataBuilder.parent.add(self).await
    → BaseTable::add (trait method)                       (table.rs)
      → impl BaseTable for RemoteTable::add               (remote/table.rs)
        → encode batch into Arrow IPC bytes
        → RemoteClient.post("/v1/table/{name}/add/", ...) (remote/client.rs)
          → wrap in RetryCounter loop                     (remote/retry.rs)
            → reqwest::Request → tokio + reqwest          (reqwest crate)
              → server                                    (out of scope)
            ← reqwest::Response
          ← RetryCounter handles errors / decides retry
        ← Response
        → parse JSON to AddResult
      ← AddResult
    ← AddResult
  ← AddResult
```

And on the **local** side:

```
Table::add(batch).execute()
  → AddDataBuilder.execute → BaseTable::add
    → impl BaseTable for NativeTable::add               (table.rs ~ line 2178)
      → coerce batches with WithEmbeddings if needed    (embeddings.rs)
      → write via lance::dataset::InsertBuilder         (lance crate)
        → object_store puts the file                    (object_store crate)
      → DatasetConsistencyWrapper::update(new_dataset)  (table/dataset.rs)
    ← AddResult
```

These are the two completely different code paths that the `BaseTable`
trait abstracts away. Internalize this picture.

## 1. The shape — thin Rust, thick wrapper

Both Python and Node bindings follow the same shape:

```
            ┌──────────────────────────────────┐
User code ─►│ Pure Python / TypeScript wrapper │   <-- thick: niceties,
            │ python/python/lancedb/...        │       errors, type
            │ nodejs/lancedb/...               │       conversions, helpers
            └──────────────────────────────────┘
                          │
                          ▼  FFI call into native code
            ┌──────────────────────────────────┐
            │ Rust binding crate               │   <-- thin: just exposes
            │ python/src/...                   │       lancedb-core types
            │ nodejs/src/...                   │       to Python/JS
            └──────────────────────────────────┘
                          │
                          ▼  Plain Rust call
            ┌──────────────────────────────────┐
            │ rust/lancedb (the core crate)    │
            └──────────────────────────────────┘
```

The Rust binding layer is thin because it's the boring boilerplate (type
conversions, runtime management). The wrapper layer is thick because it
adds language-native niceties — sync APIs in Python, Promise chains in JS,
better errors, helper functions.

---

```rust
pub fn _lancedb(_py: Python, m: &Bound<'_, PyModule>) -> PyResult<()> {
```

```rust
#[pyclass]
pub struct Table {

    pub fn name(&self) -> &str { self.inner.name() }
```

The patterns are:

* **`#[pyclass]`** on a struct exposes it as a Python class.
* **`#[pymethods] impl T { ... }`** declares its Python-visible methods.
* For sync/cheap methods (`name`, `is_open`), just write a normal Rust fn.
* For async work, **use `future_into_py(py, async move { ... })`** to
  convert a Rust `Future` into a Python `awaitable`. This is what gives you
  Python's `await` syntax on the other side.
* PyO3 conversions: `arrow::pyarrow::ToPyArrow` / `FromPyArrow` to bridge
  Arrow ↔ pyarrow (the Python side uses pyarrow types).
* Errors: `PyResult<T> = Result<T, PyErr>`. Conversion `lancedb::Error →
  PyErr` lives in `python/src/error.rs`.

## 3. The async-from-sync trick

LanceDB's Rust core is async. Python users want both `async def` *and*
plain sync methods. So PyO3 gives you `awaitable`s; the Python wrapper
layer also exposes a sync version that drives the awaitable to completion
on a background event loop.

Two pieces make this work:

### The Rust side: a fork-safe Tokio runtime

Open `python/src/runtime.rs` (~150 lines) and read the top doc-comment.

```rust
//! Fork-safe wrapper around tokio + pyo3-async-runtimes.
//!
//! `pyo3_async_runtimes::tokio` keeps its multi-threaded runtime in a
//! `OnceLock` that can never be replaced.  Tokio's worker threads do not
//! survive `fork()`, so once a child inherits a "frozen" runtime, every
//! `future_into_py` call hangs forever.
//!
//! We sidestep the global by routing every future through our own
//! [`LanceRuntime`] backed by an [`AtomicPtr`] to a tokio runtime that we
//! own.  A `pthread_atfork` child handler nulls the pointer; the next
//! `spawn` rebuilds the runtime in the child.
```

That comment tells you everything you need to know. The takeaway: this is
hard, has been gotten wrong in the past, and the file deserves respect when
you change it.

### The Python side: `BackgroundEventLoop`

Open `python/python/lancedb/background_loop.py` (~50 lines):

```python
class BackgroundEventLoop:
    def __init__(self):
        self.loop = asyncio.new_event_loop()
        self.thread = threading.Thread(
            target=self.loop.run_forever,
            name="LanceDBBackgroundEventLoop",
            daemon=True,
        )
        self.thread.start()

    def run(self, future):
        concurrent_future = asyncio.run_coroutine_threadsafe(future, self.loop)
        return concurrent_future.result()

LOOP = BackgroundEventLoop()
```

A daemon thread runs an asyncio loop. Sync methods on `LanceTable`:
1. Call the **async** method on the underlying `AsyncTable` (which is the
   PyO3-exposed `_lancedb.Table`).
2. Drive the resulting coroutine on `LOOP`.
3. Return the result.

So Python users see two parallel APIs (`Table` sync and `AsyncTable` async),
both backed by the same Rust core.

## 4. The Python wrapper layer — `python/python/lancedb/`

Open `python/python/lancedb/__init__.py`:


The package re-exports user-facing classes, sourcing from `db.py`,
`table.py`, etc.

`table.py` is **190 KB** — it's the big one. It defines:

* `class Table(ABC)` — abstract base class. Defines abstract methods.
* `class LanceTable(Table)` — sync local impl. Each method calls
  `LOOP.run(self._async_table.method(...))`.
* `class AsyncTable` — async impl. Each method `await`s the Rust-exposed
  PyO3 `_lancedb.Table` method.
* The `RemoteTable` Python class (in `python/python/lancedb/remote/table.py`)
  inherits from `Table` and routes through the same Rust binding.

Conventions:

* Type hints everywhere. The `_lancedb.pyi` file declares stubs for the
  Rust-implemented classes so editors get autocomplete.
* `async_def` mirrors of every sync method (the `AsyncTable` class).
* `pydantic.py` provides a Pydantic ↔ Arrow bridge so you can pass
  `pydantic.BaseModel` subclasses where LanceDB expects an Arrow schema.
* Error mapping: Rust `Error::TableNotFound` → Python `KeyError` etc., set
  up in `python/src/error.rs`.

The Python wrapper is where most **embedding** and **reranker**
implementations live — see `embeddings/` and `rerankers/`. The Rust core
only ships the three big ones (OpenAI, Bedrock, sentence-transformers).

## 5. napi-rs anatomy — `nodejs/src/`

[napi-rs](https://napi.rs/) is the analog of PyO3 for Node.js. Same shape:
expose Rust types to JS via a native addon.

Open `nodejs/Cargo.toml`:

```toml
[lib]
crate-type = ["cdylib"]
name = "lancedb"

[dependencies]
napi = "..."
napi-derive = "..."
lancedb = { path = "../rust/lancedb" }
```

Open `nodejs/src/lib.rs`:

```rust
#[napi(object)]
pub struct ConnectionOptions { ... }
#[napi(object)]
pub struct OpenTableOptions { ... }

#[napi_derive::module_init]
fn init() {
    let env = Env::new().filter_or("LANCEDB_LOG", "warn");
    env_logger::init_from_env(env);
}
```

`#[napi(object)]` on a struct makes it visible as a JS object/interface
(a plain interface, not a class). `#[napi]` on a struct makes it a JS
class. `#[napi]` on a method exposes it.

Open `nodejs/src/table.rs` (~28 KB):

```rust
#[napi]
pub struct Table {
    pub name: String,
    pub(crate) inner: Option<LanceDbTable>,
}

#[napi]
impl Table {
    #[napi]
    pub fn display(&self) -> String { ... }

    #[napi]
    pub fn is_open(&self) -> bool { self.inner.is_some() }

    #[napi]
    pub async fn add(&self, ipc_buffer: Vec<u8>, mode: String) -> napi::Result<...> {
        let reader = ipc_file_to_batches(ipc_buffer)?;
        let result = self.inner_ref()?.add(reader).mode(...).execute().await?;
        Ok(...)
    }
    ...
}
```

Patterns:

* **napi-rs supports `async fn` directly** — a Rust async fn becomes a JS
  function returning a `Promise<T>`. No futures-into-py dance needed.
* Data goes across the FFI as **byte buffers** containing **Arrow IPC**
  (see `ipc_file_to_batches`, `schema_to_buffer` in `nodejs/src/util.rs`
  and `nodejs/src/iterator.rs`). This is fast and language-neutral —
  same bytes Python and Java use.
* napi-rs auto-generates `index.d.ts` (TypeScript declarations) at build
  time. Run `npm run build` in `nodejs/` to regenerate.

   add one. Look at any `BaseTable` method and check the `#[cfg(test)]`
   block at the bottom of `table.rs`.
3. **Adding a missing builder method** — many builders (e.g.
   `IvfPqIndexBuilder`) have parameters set internally but no setter on
   the public type. Adding one is small, easy to test, and visible.
4. **Improving an error message** — `error.rs` variants like
   `EmbeddingFunctionNotFound` could include the list of available
   functions in the registry to help debugging. Tiny PR, big quality
   improvement.
5. **Cross-language parity** — find a Rust core method missing on Python
   or TypeScript. Use `AGENTS.md`'s recipe to add it.

Avoid as your first PR:

* Anything in `rust/lancedb/src/remote/table.rs` (huge, depends on the
  cloud server's API contract).
* Anything in `lance` itself — that's a separate repo with stricter
  review.
* Performance optimizations — they need benchmarks and the project will
  push back hard on un-benchmarked claims.

---

## When stuck — patterns

* **"This async function returns a future I can't `.await`."** You
  forgot `.await`, or you're inside a non-async function. Wrap it with
  `tokio::runtime::Runtime::new().unwrap().block_on(...)` for a one-off,
  or refactor the caller to be `async`.
* **"The compiler says `T` doesn't implement `Send`."** Something in
  your function holds a non-`Send` value across an `.await`. Common
  culprits: `Rc<...>`, `RefCell<...>`. Switch to `Arc<...>` /
  `Mutex<...>` or `parking_lot::Mutex<...>`.
* **"My PyO3 changes don't take effect."** Did you run `make develop`?
  PyO3 produces a `.so` / `.pyd` — Python imports the cached version
  unless you rebuild.
* **"napi-rs says my type is missing a `JsObject` impl."** Try a simpler
  shape: `Vec<u8>` (becomes `Buffer`), `String`, `Vec<String>`,
  `HashMap<String, String>`, or a struct annotated with `#[napi(object)]`.
  Don't try to expose a complex Rust struct directly.
* **"Tests are slow because of the embedded server in `--features
  remote`."** The remote tests use a mocked HTTP handler (see
  `python/src/error.rs` `infer_error` and the `Table::new_with_handler`
  test helpers in `rust/lancedb/src/table.rs`). Most of the time you
  don't need a real server.

## Concepts (LanceDB-specific)

| Term | Meaning |
|------|---------|
| **Table** | A directory of `.lance` files. Schema, fragments, manifests, indices. |
| **Database / Catalog** | A container of tables. Backed by `ListingDatabase`, `LanceNamespaceDatabase`, or `RemoteDatabase`. |
| **Namespace** | A scoping unit within a database (sub-catalog). Empty `Vec<String>` = root. |
| **Connection** | A user-facing handle = `Arc<dyn Database>` + embedding registry. |
| **NativeTable** | The local Lance-backed `BaseTable` impl. |
| **RemoteTable** | The HTTP-backed `BaseTable` impl. |
| **Manifest** | A versioned snapshot of a table's metadata. Each commit produces a new manifest. |
| **Fragment** | A unit of physical storage within a Lance dataset. Many rows per fragment. |
| **Vector column** | A column with `DataType::FixedSizeList<Float16/Float32>`. |
| **IVF** | Inverted File index: partition vectors into Voronoi cells. |
| **PQ** | Product Quantization: compress vectors via per-subvector codebooks. |
| **HNSW** | Hierarchical Navigable Small World: graph-based vector index. |
| **FTS** | Full-Text Search: BM25 over text columns. |
| **Hybrid search** | Vector + FTS combined and reranked. |
| **RRF** | Reciprocal Rank Fusion: default reranker for hybrid search. |
| **nprobes** | How many IVF cells to inspect during a vector search. Higher = slower + more accurate. |
| **refine_factor** | Multiplier on candidates pulled before final ranking. |
| **prefilter / postfilter** | Whether to apply scalar filters before or after vector search. |
| **Embedding function** | Maps domain data (text, image) → vector. Configured per-column. |
| **EmbeddingDefinition** | The `(source_column, dest_column, function_name)` triple stored in table metadata. |
| **Read consistency interval** | How stale a cached `Dataset` can be before refreshing. |
| **Tag** | A human-readable name pointing to a specific table version. |

## Concepts (general)

| Term | Meaning |
|------|---------|
| **Arrow** | Columnar in-memory format. `RecordBatch` is the unit of data movement. |
| **RecordBatch** | A "page" of rows: schema + Vec of column arrays. |
| **DataFusion** | Embeddable SQL/DataFrame engine in Rust. Used for filters, projections, exec plans. |
| **ExecutionPlan** | A DataFusion physical plan tree. Streams `RecordBatch`es when run. |
| **object_store** | Crate abstracting S3/GCS/Azure/local FS uniformly. |
| **Tokio** | Async runtime. Drives `Future`s. |
| **Stream** | Async iterator. Pulled with `.try_next().await` or collected with `.try_collect()`. |
| **PyO3** | Library for writing Python extensions in Rust. |
| **napi-rs** | Library for writing Node.js native addons in Rust. |
| **JNI / jni-rs** | Java native interface. Used by `java/` bindings. |
| **Arrow IPC** | Wire format for Arrow data. Used over HTTP and FFI. |
| **BM25** | Classic full-text relevance scoring (used by Lance's FTS). |
